Q1 · Loader 层：为什么既有 chunk marker 又有 prop marker？

Loader 层：Source → chunkText() → chase 写入 chunk marker，然后命题提取层：LLM 拆分 → prop marker。为什么刚输入时要切 chunk，之后又要切命题？是否有更好的做法？

现状（从源码提取）

步骤谁执行标记格式是否需要 LLM
1. 切 chunkchunkText() in src/ingest/loader.ts<!-- chunk:1 -->…<!-- /chunk:1 -->❌ 纯机械
2. 写 chaseKnowledgeStore.saveRaw() in src/knowledge/store.ts同上 + frontmatter❌ 纯机械
3. 切命题extractPropositions() in src/ingest/proposition.ts<!-- prop N -->✅ deepseek-v4-flash
4. 编译runCompilePass() in src/cli/commands/compile-from-props.tswiki node frontmatter propRefs: ["N"]✅ deepseek-v4-flash
5. 审计auditWiki() in src/knowledge/audit.ts验证 propRefs 在 chase 中存在结构: ❌ / 语义: ✅

chunkText() 具体行为（从源码）：

  • 按 \n\n+ 分割段落
  • 贪心合并段落，直到累计 ~2000 tokens
  • 每个 chunk 末尾保留 overlap tokens 的哨兵（与下一 chunk 开头重叠）
  • 目标：确保 chunk 边界不会切断关键上下文

prop marker 插入点：LLM 输出 {"propositions": [{"index":1, "text":"…", "chunkIndex":1}]}，代码遍历每个命题，找到对应 chunk 的 <!-- /chunk:N --> 关闭标记后插入 <!-- prop 1 -->。

关键桥接：collectPropIndices() 的 fallback——如果 chase 文件有 chunk marker 但没有 prop marker（命题提取还没执行），collectPropIndices() 回退到 collectChunkIndices()。这意味着系统在命题提取前也能运作（粗粒度检查）。

为什么是两层标记而不是一层？

根因：chunk 和 prop 解决的是两个完全不同的问题。

ChunkProp
目的控制 LLM 上下文窗口大小控制语义审查粒度
边界机械的（token 数 + 段落）语义的（原子事实）
产生者代码（chunkText，0 cost）LLM（一次 deepseek 调用）
粒度~2000 tokens（几十个事实混在一起）1 个自包含的原子事实
为什么不能跳过LLM 有 context window 限制。30 页 PDF → 5万 tokens → 必须在机械层先切成 ~2000 token 块才能喂给 LLM。段落级 chunk 会引入无关上下文。"claim X 是否被 chunk 2 支持？"——chunk 2 可能包含 15 个不相关的事实，干扰 LLM judge 的判断。

EMNLP 2024 (Dense X Retrieval) 实验证据：命题级分块在 Recall@20 上比段落级提升 2-10%。原子命题 = 自包含语义单元 = 没有上下文溢出 = LLM judge 更准确。

是否有更好的做法？

方案 A：跳过 chunk，直接用 LLM 从原文提取命题（chunk-free）。

❌ 不可行。30 页 PDF → 5万+ tokens → 超出 LLM context window。你需要先把原文切成 LLM 能处理的块，chunk 是必然的。

方案 B：LLM 提取命题后，用命题边界重新切 chunk，消除 chunk marker（prop-only）。

⚠️ 理论上可行但复杂度高。需要命题提取输出文本位置（start/end offset），然后按命题边界重新分块，写回 chase 覆盖原 chunk。好处：只有一个标记系统。代价：命题提取失败时失去 fallback，且 offset 计算不稳定（LLM 输出的命题文本可能不是原文的精确子串）。

方案 C：保留两层，但优化——chunk 越小越好，prop 直接取代 chunk 作为编译输入（当前做法）。

✅ 当前做法。chunk 只为一个目的存在——让 LLM 读得动原文。一旦命题提取完成，编译和审计都以 prop 为输入，chunk 退化为纯读缓存。collectPropIndices 的 fallback 确保系统在命题提取前也能运作。

方案 D：不切 chunk，直接长上下文 LLM 一次处理（Gemini 2.5 Pro 支持 100万 tokens）。

⚠️ 技术上可行但成本高。100万 token 输入 ≈ $1.25/次（deepseek 定价），且长上下文推理质量随长度衰减。保留 chunk 作为成本优化——命题提取只需一次调用。

总结：两层标记是分层的必然结果

Layer 0 (机械)  Layer 1 (LLM)    Layer 2 (LLM)
  原始文本       命题提取         编译 + 审计
    ↓              ↓                ↓
  chunk           prop             wiki node
  marker          marker           frontmatter.propRefs
  (机械切分)      (语义切分)       (证据链)

chunk 解决 "LLM 能不能读"，prop 解决 "judge 能不能精确判断"。两个问题分属不同层，合并会导致其中一层妥协。

Q2 · DeepSeek 100万上下文了为什么还要切？切完再提取 prop 会影响语义吗？

DeepSeek 现在的上下文窗口已经有 100万 tokens 了，为什么 loader 层还要切 chunk？而且切完 chunk 之后再做命题提取，会不会破坏语义完整性——LLM 看到的不再是一个连续文档，而是一堆被 标记打断的片段？

源码核实结果

事实 1：命题提取不逐 chunk 发送。

extractPropositions(chaseContent, llmCaller) 的 buildPropositionPrompt() 直接把 ${chaseContent} 拼进 prompt——全文一次发送。不是每个 chunk 调一次 LLM。

function buildPropositionPrompt(chaseContent: string): string {
  return `请将以下文档拆分为原子命题。输出 JSON 对象……

---
${chaseContent}   ← 完整 chase 内容，所有 chunk 在一起
---`;
}

事实 2：e 基本画像实测——chase 文件 37156 chars ≈ 10616 tokens → 完全在上下文窗口内。

事实 3：compile 阶段有限制——propTexts 截断到 12000 字符。

const propTexts = props.map(p => `[prop ${p.index}] ${p.text}`).join('\n\n');
${propTexts.slice(0, 12000)};   // ← 这里截断！

这是真正的瓶颈：命题提取可以处理全文，但编译时 LLM 只看到前 12000 字符的命题。对 30 页论文（~50000 tokens），大部分命题会在编译阶段丢失。

事实 4：extract-props CLI 用的 model 不一致。

调用位置modelmaxTokensthinking
extract-props CLI"deepseek-chat" (硬编码)16000未设置 ❌
pipeline.tsconfig.model (来自 .env)32768disabled ✅

切 chunk 是否影响语义理解？

当前实现：不影响。LLM 看到的是完整文档——chunk 标记只是内联注释，不是分隔符。

LLM 实际读到的内容（简化版）：

<!-- chunk:1 id=... -->
段落一的完整文本内容……
<!-- /chunk:1 -->
<!-- prop 1 -->1/e约等于0.3679…<!-- prop 2 -->极限lim_{n→∞}(1-1/n)^n = 1/e…
<!-- chunk:2 id=... -->
段落二的完整文本内容……
<!-- /chunk:2 -->

LLM 能看到全部文本的连续性。chunk marker 和 prop marker 对它来说就像 HTML 注释——不破坏语义流。实验证据：e 基本画像的 104 个命题质量正常（人工抽样确认）。

那为什么还要切 chunk？不是什么用都没有吗？

chunk 存在的原因不是语义层面，是工程层面。三个实际用途：

用途如果不切 chunk 会怎样
1. collectPropIndices() 的 fallback 如果命题提取还没执行（新 chase 文件刚写入），系统用 chunk indices 做粗粒度审计——保证系统在 prop 提取完成前也能运作。
2. 语义审计的 selectChaseChunks() 审计时 LLM judge 需要原文上下文。如果只送一个 prop 的文本（~20 字），judge 缺少上下文无法判断。chunk 提供了「送这个 prop 所在的整段原文」的机制。
3. 旧 ingest 管线的 chunkRefs 引用 v5 遗留代码（ingest.ts, prefix.ts）大量使用 chunkRefs。移除 chunk 需要同时重构这些文件。

如果你决定大规模输入的论文——真正需要改的是什么？

瓶颈不在 chunk，在 compile 的 12000 字符截断。

流程是：loader → chunkText(机械, ~2000t) → chase → extractPropositions(全文, 不受限) → compile(propTexts.slice(0, 12000) ← 这里丢数据)

解决方案（按优先级）：

  1. compile 去掉 12000 截断——改用模型 maxTokens 做自适应（当前 8192 输出 tokens 完全够容纳更多命题）。
  2. 如果单次 compile 装不下——分多个 compile pass：按 chunk 分组命题，每批编译产生该批的节点。最后用 graph edges 连接跨批节点。
  3. chunk 本身可以保留——它不是瓶颈，但在大论文场景下作为「分批编译的边界」是合理的工程抽象。

总结

问题答案
chunk 是否影响命题提取的语义理解？不影响。全文一次发送，chunk marker 只是内联注释。
DeepSeek 大上下文了为什么还要切？不是因为上下文限制，是因为工程需要：fallback、审计上下文选择、旧代码兼容。不是语义需要。
30 页论文的真正瓶颈在哪？compile 阶段的 propTexts.slice(0, 12000)。命题提取可以处理全文，但编译截断了。
chunk 可以去掉吗？技术上可以，但需要重构 3 个模块（ingest.ts, prefix.ts, semantic-audit.ts）。优先级低于修复 compile 截断。

Q13-1：矛盾检测产生的是 edges，不是 counter 节点

关键区分：contradicts edge ≠ counter node。

contradicts edgecounter node
是什么两个已有 wiki 节点之间的关系——「A 和 B 在同一事实上给出互斥结论」一个独立的 wiki 节点——「关于 X 的反方观点」——有完整的 claim/evidence/propRefs
谁产生daemon contradiction.ts：新节点 vs 已有节点 → LLM judge理论上应该由 LLM 从材料中提取，或由人类手动创建
存储位置frontmatter.edges → graphwiki/counters/counter-xxx.md
当前存在?⚠️ 检测运行，但不写 progress.md❌ wiki/counters/ 目录为空

矛盾检测不产生 counter 节点——它只产生 contradicts 边。这两个是不同的东西。counter 节点需要有人从材料中提取反方观点作为独立断言。矛盾检测只是标记「两个已有节点有矛盾」——它不会凭空创造新节点。

counter 节点可以来自 daemon 自进化吗？

理论上可以——但需要一个新的 daemon 步骤：

  1. contradiction.ts 检测到「节点 A 和节点 B 矛盾」→ contradicts edge
  2. 当同一个主题出现 3+ 条 contradicts edges → daemon 识别为「争议主题」
  3. Daemon 调用 LLM：总结这个争议是什么 → 生成 counter 节点
  4. Counter 节点标注 reflowOrigin: "contradiction-resolution"

但这是自进化系统尚未实现的部分——「从矛盾模式中合成 counter 节点」需要 LLM 推理，而不是简单的模式匹配。

当前最合理的 counter 来源：ingest 时 LLM 主动提取。在 compile prompt 中加一条指令：

如果材料中有与主流观点矛盾或提出反例的命题，也编译为 kind=counter 的节点。
counter 节点的 claim 应以「通常认为…但…」的形式呈现反方观点。

这比 daemon 合成更简单、更可靠——LLM 在读到材料时就能识别 counter。

Q13-2：qmd 在设计中的位置——不是遗漏，是指定不清

qmd 在原始设计中的定位：

  • 决策表 #3：「搜索后端切换。v1 只用 KeywordSearchProvider，qmd 集成时再加 QmdSearchProvider。YAGNI。」
  • 设计文档中出现 qmd 作为代码示例：class QmdSearchProvider implements SearchProvider { // BM25 + Vector + RRF + LLM Rerank }
  • 设计说：「qmd 处理搜索端（混合搜索 + 重排序），engine 处理 board 端（强制注入 + 局面装配）。两部分通过搜索结果衔接。」

不是「遗漏」——是三个关键细节在设计时未被指定：

未指定的细节设计说了什么实际应该是什么
qmd 索引什么？ 未指定。设计只说「qmd 处理搜索」 wiki/ 目录——每个 wiki 节点作为一个文档。否则 qmd 搜索文件和 board 搜索知识节点不在一个语义层
qmd 结果如何映射到 wiki 节点？ 未指定。假设「通过 SearchProvider 接口」但没定义映射规则 qmd 返回文件路径 → parseWikiFile(path) → 完整 BoardNode（含 propRefs/edges）
qmd vs keyword：什么时候切？ 「v1 用 keyword，qmd 未来加」——但没定义切换条件 当 wiki 节点量 >500 时，keyword 搜索的精度下降，qmd 的向量语义搜索开始体现优势。在此之前 keyword 足够

根因：qmd 在设计中被当作「更好的搜索」——但没有考虑它搜索的是不同层面的东西。设计假设 qmd 搜索 wiki 节点是自然成立的——但实际上 qmd 是一个文件搜索引擎，需要额外配置才能索引 wiki 节点。这个「自然成立」的假设是 gap 的来源。

Q10 · 三路搜索和 board 注入——为什么不是做同一件事？

理想态是三路并行搜索（BM25+Vector+Graph）决定排名，然后 board 再做强制注入。这两步不都是在处理 graph 对搜索结果的影响吗？为什么不是同一件事？

用一个具体查询说明区别

场景：用户问「1/e 在概率论中有什么应用？」wiki 中有 10 个 e 相关节点。

节点类型BM25VectorGraph 关系
A: 秘书问题与1/e最优策略concept0.900.85—
B: 错位排列的1/e极限concept0.850.80related→A
C: 1/e不是自然常数counter0.100.15contradicts→A
D: 指数过程的1/e时间常数concept0.750.70—
E: 泊松分布与1/e深层联系insight0.300.35derived_from→A, derived_from→B

只看三路搜索（无 board 注入）

BM25 + Vector + Graph → RRF 融合后排名：A(0.92), B(0.83), D(0.73), E(0.55), …C(0.25)

取 top 5 给 LLM：A, B, D, E 和另一个节点。

C（counter 节点）排在 #8，LLM 看不到。虽然 graph 给了它一点加分（从 0.15→0.25），但 counter 和查询的语义相似度天然低——它不会因为 graph 信号就跳进 top 5。

结果：LLM 给出了一个没有反方观点的答案——信息不完整。

只看 board 注入（无三路搜索）

BM25+Vector 搜索返回 seed：A(0.90), B(0.85), D(0.75)。
Board 强制注入：C(contradicts→A, score=0.5), E(derived_from→A,B, score=0.5)。

最终 board：A(0.90), B(0.85), D(0.75), C(0.5), E(0.5)。

问题：C 和 E 都是 0.5。但 E 被 2 个 seed 共同引用（更强的结构信号）而 C 只是一个 counter。LLM 无法从分数区分哪个更值得关注。

三路搜索 + board 注入（互补）

三路搜索后排名：A(0.92), B(0.83), D(0.73), E(0.55), …C(0.25)
Board 注入：C 仍在 top 5 之外 → 强制补入 C(0.5)

最终 board：A(0.92), B(0.83), D(0.73), E(0.55), C(0.5)

结果：E 因为被两个 seed 引用而获得了更高的自然排名（graph 信号融入了分数）。C 因为 contradicts 关系而被强制补入（board 保证了完整性）。LLM 看到所有关键节点，且分数有意义。

根本区别：graph 在两步中扮演的角色不同

三路搜索中的 GraphBoard 注入中的 Graph
作用提升排名——结构连接使节点在列表中更靠前保证存在——不管排名，这个节点必须出现
适用边类型derived_from, related, supports
（「节点 E 被 3 个 seed 引用 → E 可能很重要」）		contradicts, superseded_by
（「节点 C 和 seed A 矛盾 → C 必须被 LLM 看到」）
操作调整分数（graph 是打分信号）调整集合（graph 是包含规则）
解决的失败模式「应该排名靠前的节点排到了后面」「应该出现的节点根本没出现」
对 counter 节点的效果微弱提升（counter 和查询语义相似度天然低，graph 信号无法完全补偿）强制包含（counter 必须出现，不管相关性多低）
对 related 节点的效果显著提升（被多个 seed 引用的 related 节点天然有结构重要性）无必要（related 节点通过三路搜索已经被自然排在前面了）

类比：搜索引擎 vs 编辑审阅

三路搜索 = Google 的 PageRank。 Graph 结构（谁链接到谁）影响结果的排名。被很多相关页面链接的内容排名更高。

Board 注入 = Wikipedia 编辑的「中立性」检查。 不管搜索排名如何，反方观点必须出现。这不是排名问题——是内容完整性要求。

两者回答了不同的问题：

  • 三路搜索：「在符合查询的结果中，谁应该排最前面？」—— graph 是打分信号
  • Board 注入：「在 LLM 做推理前，哪些信息必须出现在它面前？」—— graph 是包含规则

Graph traversal 的精简实现

agentmemory 用 Dijkstra BFS + 边权重 + 路径长度——因为它的图有万级节点和加权边。你的图：10 节点，1 边/节点。

// 精简版——现在就够用
function walkGraph(seedIds: string[], graph: GraphData): Map<string, number> {
  const scores = new Map();
  for (const seedId of seedIds) {
    for (const edge of graph.edges) {
      if (edge.from === seedId || edge.to === seedId) {
        const other = edge.from === seedId ? edge.to : edge.from;
        const boost = edge.type === "contradicts" ? 0.3
                    : edge.type === "derived_from" ? 0.2 : 0.1;
        scores.set(other, (scores.get(other) ?? 0) + boost);
      }
    }
  }
  return scores;
}

不需要 Dijkstra。不需要路径长度。1-hop + 边类型加分。

// 照抄 agentmemory hybrid-search.ts 第 216-218 行
const K = 60;
function rrfFusion(
  bm25Ranks: Map<string, number>,
  vectorRanks: Map<string, number>,
  graphScores: Map<string, number>,
): Map<string, number> {
  const allIds = new Set([...bm25Ranks.keys(), ...vectorRanks.keys(), ...graphScores.keys()]);
  const scores = new Map();
  for (const id of allIds) {
    const br = bm25Ranks.get(id) ?? Infinity;
    const vr = vectorRanks.get(id) ?? Infinity;
    const gs = graphScores.get(id) ?? 0;
    scores.set(id,
      1.0 / (K + br) +     // BM25: 关键词 = 最重要
      0.8 / (K + vr) +     // Vector: 语义 = 次重要
      0.4 * gs             // Graph: 结构加分（已在上游归一化）
    );
  }
  return scores;
}
