为什么边界画在 chase 入口：loader 层是可替换的——以后加 EPUB/DOCX loader 只需遵守合同（产出纯文本），发动机不用改。命题提取由 daemon 统一处理——不在 loader 层调 LLM。这个设计让 loader 层可以无限扩展而不影响内核。

命题提取 Prompt 架构

维度设计
模型deepseek-v4-pro（深度理解任务，需要语义判断命题边界）
输入chase 纯文本（loader 产出的 Markdown）
输出命题列表——每个命题是自包含的原子事实，编号从 1 开始
缓存一次性调用，结果持久化在 chase MD 中。compile 和 audit 复用同一编号
失败处理daemon 写状态报告（progress.md），agent 读到后决定是否重试或人工复核

无向后兼容

当前 wiki 节点量少——全部清空后直接用新格式。chase.ts 只支持 <!-- prop N --> marker，不支持旧 <!-- chunk N -->。frontmatter 只用 propRefs，不支持 chunkRefs。代码简洁——无 fallback 逻辑、无双字段共存、无迁移脚本。

为什么命题级：EMNLP 2024 (Dense X Retrieval) 实验证据：命题级分块在 Recall@20 上比段落级提升 2-10%。judge 对比 "claim X 是否被 prop 7 支持" 时，输入是自包含原子事实，没有无关上下文干扰——减少 false pass 和 false fail。

ingest = node + edge 的双重产出。

每次 ingest（编译管线）完成后，系统自动产生两类 edge：

  • derived_from — compile LLM 在提取节点时标注推理链（如「节点B的claim基于节点A的命题」）→ 直接写入 frontmatter，无需人类确认
  • related — compile LLM 自动推导语义关联（如同一 source、共享 tags、claim 关键词重叠）→ 直接写入

这两类 edge 是 ingest 的内置产出——不需要额外步骤。contradicts 和 superseded_by 由 daemon 在 compile 后异步检测，写入 progress.md 等待人类确认。

inspire 是独立的探索工具——它的目的是降低发现阈值，帮助人类发现不明显、非线性的关联和因果连接。inspire 产出的 related 候选必须人类确认（因为 LLM 在 inspire 模式下做的是推测性推理，不是在编译过程中根据命题原文推导）。

Exit Code 语义

exit code含义agent 行为
0成功解析 JSON 输出，继续下一步
1运行时错误停止，报告 error
2结构审计硬关卡失败停止，不可继续 query。failure JSON 中 stage=audit-structure

⚠️ 关键前提：自进化的起点是 wiki 节点，不是原材料。

重新 ingest 同一篇材料是人类的主动决策（CLI ingest2），不是 daemon 的自动化任务。覆盖式 cleanup 确保旧节点被替换。

五个方向详解

取代

新材料淘汰旧声明

矛盾检测 → contradicts 边 → 旧声明标候选 superseded → 人类确认后 superseded_by 写入 frontmatter。board 跳过 superseded 节点。

自愈

定期 lint + 分级修复

机械问题（断链、图重建）自动修复。语义问题（claim 过时、内容漂移）生成工单 → 人类确认后手动修复。

连接

发现隐藏关联（两层机制）

第一层 · ingest 自动：每次 compile 时，LLM 自动标注 derived_from 边（推理链）和 related 边（语义关联）→ 直接写入 frontmatter。这是 ingest 的内置产出——不需要额外步骤。

第二层 · inspire 探索：人类手动触发 inspire → 装配 inspire 模式 board → 产出 edge 候选。inspire 的目的是降低发现阈值——帮助人类发现不明显、非线性的关联和因果连接。LLM 在 inspire 模式下做推测性推理，因此产出必须人类确认 → 写入 frontmatter edges。

缺口

标记知识空白

query 的 missingEvidence + inspire 的 missingEvidence + lint 的孤立节点 → 写入 progress.md → 人类决定是否填补。

强化

新材料支持已有声明

compile 后 daemon 检测新节点与已有节点的语义一致 → 双维度判断（一致度 + 增量度）→ 增量度 > 阈值 → 标记候选 → 人类确认后 auditScore 提升。

强化检测的双维度判断

维度问题计算阈值
语义一致度两个 claim 真的在说同一件事吗？LLM judge 对比 claim A 和 claim B 的语义相似性> 0.7 才继续
证据增量度新来源是否提供了独立的经验证据？LLM judge 判断新来源是否有旧来源没有的验证> 0.5 才标记候选
trivial vs genuine reinforcement：两本书都说"1/e ≈ 0.368"——语义一致度 1.0，但证据增量度 0.0（数学常数不需要独立验证）→ 不标记候选。论文 A 说"X 有效"，论文 B 说"我们在独立实验中复现了 X"——一致度 0.9，增量度 0.8 → 标记候选。

人类确认的用户界面

人类确认时看到的最小信息集（以强化确认为例）：

### 强化候选 [reinforce-001] 已有节点: shor-rsa-threat claim: "Shor算法对RSA加密构成根本性威胁" 当前 auditScore: 0.91 chase 命题锚点: prop 2 "Shor算法对RSA加密构成威胁" 新来源节点: shor-rsa-confirmation claim: "独立实验确认Shor算法在多项式时间内分解RSA-2048" source: raw/pdf/another-paper chase 命题锚点: prop 5 "实验复现了分解结果" daemon judge 判断: 语义一致度: 0.88 (两个 claim 在说同一件事) 证据增量度: 0.75 (新来源提供了独立实验验证) 建议新 auditScore: 0.94 人类判断: ☐ 确认提升 (genuine reinforcement) ☐ 拒绝 (trivial reinforcement — 无独立验证) ☐ 跳过 (需要更多信息)
知识策展不是通知驱动的碎片化操作。认知科学上叫"注意力切换成本"——每次切换 15-25 分钟恢复。一周集中处理 20 条 vs 每天被打断 3 次——前者的总认知成本远低于后者。代码 review、论文审稿、合规审计都是批量做的——知识策展也一样。
