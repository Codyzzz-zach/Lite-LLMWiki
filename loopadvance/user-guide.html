01 / What It Is

产品是什么

LiteWikiagent 是一个 agent 无关的可演化知识层。不是 RAG，不是笔记工具，不是搜索引擎。

它把原始材料编译成带审计信号的结构化 wiki，给任何 agent 提供统一的知识操作基础设施——编译、审计、局面、维护。agent 可以换，知识层的状态继承下来，wiki 随使用越来越准。

发动机的三个组件

组件做什么为什么不可缺
编译将 chase 命题提取为 9 种 kind 的 wiki 节点没有编译就没有 wiki。必要但非壁垒——agent 自己也能调 LLM 提取节点

wiki + 图谱 = 棋盘。6 种 board 模式 = 残局的摆法。LLM = 下棋高手。发动机的核心价值不是"把文档变成文本"（谁都能做），而是"给 LLM 摆出一个经过审计的残局"（所有竞品都没做）。

02 / Why Use It

为什么要用 LiteWikiagent

如果你遇到过以下任何一个问题，LiteWikiagent 就是为你的场景设计的。

场景 A：RAG 的痛点

"我有一堆论文和笔记，每次问 LLM 问题它都要重新检索、重新拼接碎片。跨 5 篇文档的综合推理每次都是从零开始。知识不积累。"

LiteWikiagent 怎么解决：编译一次，持续复合。wiki 是持久化资产——跨引用已存在、矛盾已被标记、综合结论已反映所有已读材料。query 读的是已编译的结构，不是每次重新检索。

场景 B：LLM 幻觉的痛点

"LLM 给我的回答看起来很专业，但我不确定它是在说原文的事实还是在自己编。我不知道哪些可以信任。"

LiteWikiagent 怎么解决：每个 claim 都有命题级证据锚点。LLM judge 对比 claim 与 chase 命题原文，给出 5 维度 verdict + 可信度评分。你不信任"LLM 说的"，你信任"经过审计的 claim"——而且审计信号本身也是诚实的（不假装客观）。

场景 C：知识腐化的痛点

"我的知识库越来越臃肿。有些旧笔记已经过时了但我不知道是哪些。新笔记和旧笔记有矛盾但没人发现。孤立的知识点越来越多。"

LiteWikiagent 怎么解决：daemon 定期跑 lint——检测孤立节点、断裂引用、过时声明、缺失概念。矛盾被自动标记（contradicts 边），人类确认后取代。缺口被主动写入 progress.md。wiki 不会腐化——它会越来越准。

场景 D：Agent 换代的痛点

"我从 Claude Code 换到 Codex，之前积累的知识全丢了。每个 agent 都要我重新喂材料。"

LiteWikiagent 怎么解决：agent 无关。知识层的状态（审计信号、待确认队列、图谱关系）全部继承。换 agent 不换知识层。engine 是 agent 的外挂知识基础设施，不是特定 agent 的工具。

与 RAG / 笔记工具 / 搜索引擎的区别

传统 RAG笔记工具搜索引擎LiteWikiagent

这个产品不是给所有人的。

如果你有以下情况，LiteWikiagent 可能不是适合你的工具：

  • 你的材料不值得反复查阅。发动机每次编译的成本（LLM 调用）需要被多次查询摊薄。如果你只问一次就再也不碰那份材料了——RAG 更适合你。
  • 你不愿意每周投入 15-30 分钟做深度确认。确认清单是信息卫生的闸门。如果你会随手全点"确认"——wiki 会累积噪音，不如不用。
  • 你只想要"一键搜索 + AI 回答"的体验。搜索引擎或 ChatGPT 文件上传已经很好了——发动机需要你参与策展。

04 / Three Roles

三角色分工：daemon · agent · 人类

核心原则：daemon 是 CI 管道，agent 是决策者，人类是主编。

⚙️

daemon — CI pipeline

类比：CI pipeline——自动跑测试但不决定修不修 bug。

职责：监听文件自动命题提取、自动结构审计、定期跑 lint（只修复机械问题）、接收人类 inspire 命令（只产出候选）、回流候选标记。

不做：做语义判断、写回 wiki 内容、确认回流节点、确认取代关系、写入 edges——任何涉及"这个对不对"的决策。

🤖

agent — 开发者 + 运维

类比：开发者——读 CI 报告，决定优先级。

职责：调用 CLI 做 compile、query、inspire。读取 daemon 状态文件，自己判断优先级——要不要处理这个矛盾、要不要告诉人类有回流候选。在 wiki 局面上做推理。每次会话开头读待确认清单，帮人类做预筛选。

不做：替人类确认回流、替人类确认 edges——这些决策需要人的判断力。

👤

人类 — 主编

三件事：选材、终判、升级。定期批量处理待确认队列——不碎片化打扰。

不做：日常操作——这些是 agent 和 daemon 的事。

人类确认的批量处理：

agent 预筛选（按风险+置信度排序）+ 积压上限自动降级（<20 正常，20-50 提醒，>50 降级）。

09 / OKF Export & Import

OKF 生态兼容：导出与导入

LiteWikiagent 的 wiki 节点格式天然兼容 Google Cloud 的 Open Knowledge Format (OKF) v0.1——零改造成本的知识交换。

OKF 是一个用纯 Markdown + YAML frontmatter 表示知识的开放格式。你的 wiki 节点有 9 项 OKF 要求中 8 项零差距、1 项小映射（kind → type）。

导出为 OKF

# 将整个 wiki/ 导出为 OKF bundle
llmwiki export --okf --out ./my-kb-bundle

# 按来源/类型过滤导出
llmwiki export --okf --source raw/pdf/my-paper --out ./paper-kb

# 导出产物：一个标准的 OKF bundle 目录
# 别人不需要装 LiteWikiagent——任何 OKF 兼容工具都能读

从 OKF 导入

# 导入外部 OKF bundle
llmwiki import --okf ./external-bundle

# 预检（dry-run，不实际导入）
llmwiki import --okf ./external-bundle --dry-run

导入时每个 OKF concept 视为带完整 frontmatter 的 wiki 节点 draft——进入正常的编译管线，走 audit + gate 流程。外部内容不会绕过质量闸门。

战略价值：

LiteWikiagent 是唯一带语义审计的 OKF 兼容编译器。你的知识可以导出给任何人——对方的系统能看到审计信号、证据锚点。

07 / Scenario: Ingest

场景一：导入材料

把原始材料编译成可审计的 wiki。这是发动机的第一个组件。

flowchart LR
  RAW["raw/original/"] -->|loader| CHASE_TEXT["chase 纯文本"]
  CHASE_TEXT -->|daemon 命题提取| CHASE_PROP["chase 命题层
(带 prop marker)"]
  CHASE_PROP -->|compile Pro两步| WIKI["wiki 节点 (9 kind)"]
  WIKI -->|自动| AUDIT["audit"]
  AUDIT -->|写回| WIKI
  AUDIT -->|写| GATE["audit-gate.json"]

命题是什么：命题 = 自包含的最小事实单元。不能再拆成更小的独立命题，但也不需要额外上下文就能理解。比如"Shor算法能在多项式时间内分解大整数"是一个命题，而"量子计算有很多优势"不是命题（太模糊）。命题级分块有 EMNLP 2024 论文的实验数据支撑——在精密检索任务上显著优于段落级和句子级分块。

三种格式

格式位置说明
Markdownraw/original/md/直接读取，解析 frontmatter
PDFraw/original/pdf/提取文本后分块
LaTeXraw/original/tex/<project>/解析 \input/\include，处理 .bbl

自动模式（agent 推荐）

# 自动确认 + conservative + JSON
llmwiki ingest ../raw/original/pdf/my-paper.pdf --auto --policy conservative --json

# TeX 项目文件夹
llmwiki ingest ../raw/original/tex/arXiv-paper/ --auto --json

# 带人类锚点
llmwiki ingest ../raw/original/md/notes.md --auto --anchor "1/e 的概率意义" --json

交互模式（人类推荐）

llmwiki ingest ../raw/original/pdf/my-paper.pdf

# [a] 对齐 [s] 跳过 [m] 不同角度 [a all] 批量确认

Policy 确认策略

Policy行为适合
conservative只确认 confidence≥0.7 + 有 evidence 的 fact 类型日常使用
balanced（默认）fact 类型 high/medium 都确认；soft types 需 evidence需要总结
expansive允许 insight/counter/question，需 limits头脑风暴

JSON 输出

{ "ok": true, "sourceId": "raw/pdf/my-paper-a1b2c3d4", "created": ["wiki/concepts/e-overview.md"], "audit": { "structure": {"ok":true}, "semantic": {"ok":true, "averageScore":0.91} } }
自动审查：

--auto 模式下 ingest 完成后自动执行结构 + 语义 audit。审查结果写入 audit-gate.json。

08 / Scenario: Audit

场景二：审查知识

双重审查：结构审查（硬关卡）+ 语义审查（软关卡）。这是发动机的第一壁垒。

结构审查（无 LLM，硬关卡）

llmwiki audit --json

检查 nodeId、kind、sourceChase、chunkRefs、Claim、Evidence。不合格 → 阻止，exit 2。

语义审查（LLM judge，软关卡）

# 全量
llmwiki audit --semantic --json

# 按来源
llmwiki audit --semantic --source raw/pdf/my-paper --json

# 单节点
llmwiki audit --semantic --node one-over-e-unification --json
维度问题结果
supportClaim 是否被命题支持？aligned / stretched / unsupported
addition是否加入原文没有的主张？none / minor / major
inference是否把推理标成事实？ok / warning / failed
limits是否遗漏限制条件？ok / warning / failed
citationchunkRefs 是否覆盖关键证据？ok / warning / failed

三层防御

  • 第一层（零额外成本）：evidence-anchored 输出——judge verdict 必须带原文锚点
  • 第二层（图谱免费信号）：矛盾检测交叉验证——contradicts 边的节点至少一个 audit 有问题
  • 第三层（按需付费）：跨模型二次审计——auditScore>0.9 且频繁引用的核心 claim 用不同模型重审

软关卡 + 诚实声明

flowchart TD
  AUDIT["audit 完成"] --> WRITE["写回 frontmatter"]
  WRITE --> GATE["写 audit-gate.json"]
  GATE --> QUERY["query / inspire"]
  QUERY --> CHECK{"关卡检查"}
  CHECK -->|"结构失败"| BLOCK["❌ 阻止 (exit 2)"]
  CHECK -->|"语义失败"| WARN["⚠️ 警告放行"]
  CHECK -->|"全通过"| PASS["✅ 放行"]
  WARN --> HONEST["board 中带 auditStatus
LLM 能看到可信度"]
  PASS --> HONEST
诚实声明：

不假装 LLM judge 是客观度量。审查信号是可信度辅助信号，不是最终判决。信任模型是"让错误可见"而不是"消灭错误"。

09 / Scenario: Query

场景三：查询 — Board 残局

6 种 mode = 6 种残局摆法。搜索负责相关性，强制注入负责完整性。

竞品的查询是"搜索 → 找节点 → 扔给 LLM"。问题是搜索有盲区——counter、limit、tension 节点可能排名靠后。发动机的方案：搜索找 seeds + 强制注入靠确定性字段确保关键棋子不遗漏 = 残局。

mode搜索强制注入适合
ask混合搜索 top seedsevidence + related + 少量 counter"这个概念是什么？"
trace搜索 seedsderived_from 链祖先 + chase 命题原文"从哪来？""给我看依据"
expand搜索 seeds所有相关 method/case/equation"还能怎么理解？"
compare搜索 seeds按 sourceId 强制分组"这两个观点有什么区别？"
challenge搜索 seedsALL counter + ALL limit"站得住吗？""哪里可能错？"
inspire搜索 seedstension 节点 + cross-kind related"给我新问题"
# 基本问答
llmwiki query "为什么 1/e 是失败概率基线？" --mode ask --json

# 来源追溯
llmwiki query "从哪来？" --mode trace --with-source --json

# 挑战审查
llmwiki query "什么条件下失效？" --mode challenge --json

# 兼容别名: exact→trace, explore→expand, counter→challenge

JSON 输出

{ "ok":true, "mode":"ask", "answer":"...", "fromWiki":[{"claim":"...","nodeId":"..."}], "modelSynthesis":[{"text":"...","basedOn":[...]}], "missingEvidence":[...] }
10 / Scenario: Inspire

场景四：启发（人类手动触发）

inspire 是人类主动触发的探索命令——不是后台自动跑的。它装配 inspire 模式的 board（含 tension 节点），生成 5 类启发项候选。

# 人类手动运行——不会自动触发
llmwiki inspire --json

llmwiki inspire --seed "1/e" --json

llmwiki inspire --node one-over-e-unification --json

# 也可以 query --mode inspire
类型含义
connection两个节点间的隐藏关联
hypothesis基于 board 张力的假设
question值得进一步探索的问题
action建议的下一步行动
missingEvidence缺失的证据或视角
信息卫生：

inspire 产出的 edges 候选不自动写入。daemon 只发现和标注——写入需要人类确认。

13 / Daemon Usage

daemon 使用

daemon 是 CI pipeline——常驻进程，自动监听 + 定时维护。

七项职责

#职责触发daemon 决定留给 agent/人类
①命题提取新 chase 纯文本调 LLM质量复核
②文件监听自动导入chase 有 prop marker触发 compile选材
③wiki 变化自动审查wiki 文件变化触发 audit结果怎么用
④定期 lint + 分级修复定时器机械问题自动修语义工单优先级
⑤启发发现人类命令触发跑 inspire、产出候选哪个连接值得追查、是否写入 edges——人类在结果中勾选确认
⑥回流候选标记定时器LLM judge 筛选是否写回
⑦强化检测compile 后标记候选是否提升 auditScore

daemon 与 agent 的关系：

14 / Agent Protocol

Agent 协议

外部 agent 的标准调用流程和 JSON 契约。

flowchart TD
  A1["1. plan --json"] --> A2["2. ingest --auto --json"]
  A2 --> A3["3. 读 daemon 状态
check audit-gate.json"]
  A3 --> A4{"structure ok?"}
  A4 -->|"否"| FAIL["停止 · 结构 audit 未通过"]
  A4 -->|"是/不存在"| A5{"semantic ok?"}
  A5 -->|"否/null"| A6["⚠️ 语义审计信号弱
agent 根据 auditStatus 自行判断"]
  A5 -->|"是"| A7["5. query / inspire --json"]
  A6 --> A7

关键区别：结构审计失败（如 nodeId 缺失）→ 硬关卡，agent 必须停止。语义审计失败（如 auditScore=0.3）→ 软关卡——agent 读取审计信号后自行判断是否继续 query，如果继续则在 board 中带上 audit 警告。

llmwiki plan ../raw/original/pdf/my-paper.pdf --json
llmwiki ingest ../raw/original/pdf/my-paper.pdf --auto --policy conservative --json

# 读 daemon 状态——检查审计是否完成、关卡状态如何
llmwiki status --json
# 或直接查关卡文件
cat wiki/audit-gate.json

# 如果 daemon 的审计已完成且通过——直接 query
llmwiki query "核心问题" --mode ask --json

# 如果 daemon 未运行或审计信号缺失——agent 可自行补充
llmwiki audit --json
llmwiki audit --semantic --json

llmwiki inspire --seed "核心概念" --json

失败 JSON（仅结构审计失败时）

语义审计不产生 failure JSON——语义失败走警告放行，信号通过 frontmatter 告知 agent。只有结构审计失败（nodeId/kind/chunkRefs/Claim/Evidence 缺失）才产生 failure JSON。stage 值必须精确：

{ "ok":false, "stage":"audit-structure", "error":"structure audit failed — required fields missing", "blockingIssues":["missing-nodeId","missing-claim"], "suggestedNextActions":["run audit --json to see details"] }
Agent 禁止行为：
  • ingest 失败后继续 query
  • 结构审计失败后（nodeId/kind 缺失）把 wiki 当可靠来源
  • 语义信号弱时不告知用户——即使不阻止 query，也必须在回答中声明"本回答包含 auditStatus=warning 的节点"
  • 把 modelSynthesis 当 fromWiki
  • 隐藏 missingEvidence
  • 替人类确认回流、edges、取代——这些需要人的判断力
15 / Troubleshooting

故障排查

常见问题与解决方案。

❌ "DEEPSEEK_API_KEY not set"

系统必须有 API key。设置环境变量或 .env 文件。

⚠️ "structure audit failed"

检查 audit --json 的 issues。常见：缺少 sourceChase、chunkRefs 为空、Evidence 缺失。

⚠️ "semantic audit failed"（警告放行）

不阻止查询。检查 dimension 和 reason。这是信号——agent 和人类根据信号决定是否信任。

ℹ️ query 被关卡阻止 (exit 2)

这是结构审查的硬关卡（nodeId/kind/chunkRefs/Claim/Evidence 缺失）。先运行 audit --json 修复结构问题。语义审计失败不阻止——警告放行，信号随 board 告知 LLM。

ℹ️ chunkRefs 越界

clampChunkRefs 会自动过滤。如果审查报 citation=failed，检查 evidence 是否引用了不存在的 chunk。
