§01 — Overview

总览：一段代码，三个角色

LiteWikiagent 是一个 agent 无关、可演进的知识层。它的核心是"知识质量"——不是 chatbot 的答案质量，而是知识本身是否忠实于原始材料、是否可追溯、是否随时间变好。

三路 RRF 融合公式（v2.2 同量纲修复）

RRF（Reciprocal Rank Fusion）将三路搜索结果融合为一个排序。v2.2 的关键修复是三路统一用 weight/(K+rank) 形式——之前 Graph 路用 0.3 × score 会导致 graph 邻居分数压过语义 seed。

combinedScore = 0.4 / (K + bm25Rank)  +  0.6 / (K + vectorRank)  +  0.3 / (K + graphRank)

其中：
  K = 60（标准 RRF 常数）
  bm25Rank = BM25 搜索结果中的排名（1-based，越小越好）
  vectorRank = Vector 搜索结果中的排名（1-based）
  graphRank = walkGraph() 返回的 score 经 scoresToRanks() 转换后的排名
  缺失路径的 rank = Infinity → 该项贡献为 0（优雅降级）

同量纲的意义（Finding 1 修复）：
  旧实现 graph 路用 0.3 × score（score 0-1），graph 邻居最高 0.3，
  而语义 seed 是 0.4/(K+1)+0.6/(K+1)≈0.016——graph 邻居会压过 seed。
  改成 0.3/(K+graphRank) 后，graph 第1名 = 0.3/61 ≈ 0.0049，
  低于 BM25 第1名（0.0066）和 Vector 第1名（0.0098）。
  graph 邻居天然排不进前几名，但能进中后段——既不泛滥也不消失。
BM25 / Vector 并行（Finding 8）：v2.2 的 buildQueryBoard 使用 Promise.all 并行发起 BM25 和 Vector 搜索。默认 keyword 后端没有 searchVector → 退化为顺序但无额外开销；qmd 后端启用时两路真并行，搜索延迟降为 max(BM25, Vector)。
优雅降级：当 KeywordSearchProvider 被使用时（默认），searchVector 不存在（接口可选），Vector 路为空 Map，贡献为 0。BM25 和 Graph 仍正常工作。权重比例保持正确——不会把 Vector 的 0.6 权重转移给 BM25。
Graph-only 邻居进入候选池（Finding 1 修复）：v2.2 让被多 seed 引用的高置信 graph 邻居能进 seedNodes——从 allParsed 中查找并转为 BoardNode（score=0），按融合分数重排。之前这些节点会被丢弃。

注入规则（board.ts:506-522）：
