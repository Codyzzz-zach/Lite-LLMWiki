02 / Loader → Proposition

消灭 chunk 标记，统一 prop-only 管线

v1.0 设计明确要求「无 chunk 标记、无 fallback」——当前代码未完全执行。 代码中的 chunk marker、collectPropIndices 的 chunk fallback、wiki-parser 的 chunkRefs 解析——都是设计违规残留。

目标 chase 格式

---
sourceId: raw_pdf_e_的基本画像_3ef24c47e940cf85
fingerprint: 3ef24c47e940cf85
---

e（欧拉数）约等于 2.718281828459045，是自然对数的底数。

<!-- prop 1 -->
1/e约等于0.3678794411，约等于36.8%。
<!-- /prop 1 -->

极限lim_{n→∞}(1-1/n)^n = 1/e。当n趋于无穷大时…

<!-- prop 2 -->
极限lim_{n→∞}(1-1/n)^n = 1/e。
<!-- /prop 2 -->

与当前格式的区别：❌ 无 <!-- chunk:N --> 标记。Prop 用开闭标记环绕正文。无 prop 时 chase 为纯文本（命题提取前中间态）。

五个改造点

#改造文件
①Loader 不再切 chunk——chase 文件 = frontmatter + 正文，无 chunk 标记knowledge/store.ts
②命题提取不变——全文一次送 LLMingest/proposition.ts
③审计上下文改用 prop 邻近窗口（±3 prop）替代 chunk 选择 已实现knowledge/chase.ts（selectPropsContext()）
④compile 硬截断改为自适应——slice(0, maxTokens × 2.5) 已实现ingest/pipeline.ts（compileInputBudget()）
⑤extract-props CLI 统一配置——config.model + thinkingDisabled: truecli/commands/extract-props.ts

关键设计决策：
• 覆盖式 ingest：同 fingerprint 的旧节点先删再写——防止 wiki 膨胀。
• fingerprint 做主键：chase 文件的 fingerprint 字段（内容哈希）是清理旧节点的权威标识——文件重命名不影响清理。

• pipeline 写 progress.md：contradict/reinforce 结果不再丢弃——写入 progress.md 等待人类确认。

组件职责分配

组件谁提供怎么用
BM25 搜索qmd SDK: store.searchLex(query)调用 qmd FTS5，拿到 BM25 排名列表
Vector 嵌入qmd CLI: qmd embed -c litewiki一次性对 wiki/ 目录生成所有节点嵌入（本地 GGUF）
Vector 搜索qmd SDK: store.searchVector(query)调用 qmd sqlite-vec，拿到 Vector 排名列表
Graph traversal自研: walkGraph(seedIds, graph)1-hop 遍历 + 边 confidence 加权（照抄 agentmemory: avgConfidence × (1/pathLength)）
RRF 三路融合自研照抄 agentmemory 公式：Σ(w_i / (K + rank_i)) · 权重对齐 agentmemory: BM25=0.4, Vector=0.6, Graph=0.3
LLM 重排（可选）qmd SDK: store.rerank(query, docs)对 RRF 融合后的 top 20-40 候选做 cross-encoder 精排
qmd 集成：
• 一次性配置：qmd collection add wiki/ --name litewiki + qmd embed -c litewiki
• 不用 qmd 的 unified search()——它内部做 2-way RRF。我们要 searchLex() + searchVector() 的原始排名，加上自己的 graph rank，三路一起 RRF。
• QmdSearchProvider 从 qmd 返回的文件路径调用 parseWikiFile() 解析真实节点（含 propRefs + edges + auditStatus）。
• 默认仍用 keyword search——qmd 作为可替换后端，接口已定义，不强制启用。
• 优雅降级（对齐 agentmemory hybrid-search.ts）：默认 keyword 后端（KeywordSearchProvider）不实现 searchVector → Vector 路传空 RankMap → RRF 中 0.6/(K+Infinity)=0，Vector 自然贡献 0 分，等效「BM25(0.4) + Graph(0.3) 两路」，相对比例保持。qmd embed 配置后 searchVector 返回真向量排名，三路才齐生效。绝不用「Vector=BM25 伪造三路」——那会让 BM25 权重实际变 0.4+0.6=1.0，扭曲 Graph 相对权重。

RRF 融合公式

// 照抄 agentmemory hybrid-search.ts 第 216-218 行——三路统一用 weight/(K+rank) 形式
const K = 60;

function rrfFusion(bm25Ranks, vectorRanks, graphRanks) {
  // bm25Ranks:  Map<nodeId, BM25 rank (1-based)>
  // vectorRanks: Map<nodeId, Vector rank (1-based)>
  // graphRanks: Map<nodeId, Graph rank (1-based，由 walkGraph score 经 scoresToRanks 转换)>
  //
  // 权重对齐 agentmemory hybrid-search.ts 第 30-32 行:
  // bm25Weight=0.4, vectorWeight=0.6, graphWeight=0.3
  // 三路同量纲：graph 第1名 = 0.3/61 ≈ 0.0049，低于 BM25 第1名(0.0066)和 Vector 第1名(0.0098)
  // → graph 邻居天然排不进前几名，但能进中后段（不泛滥也不消失）

  const allIds = new Set([...bm25Ranks.keys(), ...vectorRanks.keys(), ...graphRanks.keys()]);
  const scores = new Map();

  for (const id of allIds) {
    const br = bm25Ranks.get(id) ?? Infinity;   // 缺失路 = Infinity → 贡献 0（优雅降级）
    const vr = vectorRanks.get(id) ?? Infinity;
    const gr = graphRanks.get(id) ?? Infinity;

    scores.set(id,
      0.4 / (K + br) +    // BM25: 关键字
      0.6 / (K + vr) +    // Vector: 语义（agentmemory 中权重最高）
      0.3 / (K + gr)      // Graph: 结构（同量纲，不再用 0.3*score 避免压过 seed）
    );
  }
  return scores;
}

06 / Board Assembly

简化 Board：按 mode 分派，只有两个 mode 需要注入

Mode输入额外装配需要 board?
ask三路搜索 top K seeds无❌
trace三路搜索 seedschase excerpts（从 sourceChase + propRefs）⚠️ 轻量
expand三路搜索 seeds1-hop related（graph 搜索路自然排名）⚠️ 轻量
compare三路搜索 seeds按 sourceIds 分组 + bridge nodes⚠️ 轻量
challenge三路搜索 seeds强制注入 ALL counter + ALL limits（不管 RRF 排名）✅ 完整
inspire三路搜索 seeds降低阈值：cross-kind 节点 + 低分但连通的节点✅ 完整
三路搜索和 board 注入在原理上是冲突的。
三路搜索说「我相信排名——Graph 结构影响分数，排到后面的节点就不应该出现在 LLM 面前」。
Board 注入说「我不信任排名——有些节点排名低是搜索盲区，不是它不重要，我直接加进来」。
一个相信排名，一个绕过排名——这是根本性的设计冲突。

解决方案：按 mode 分离。不要让所有 mode 都做注入。
  • ask/trace/expand/compare → 纯三路搜索，信任排名。不注入。
  • challenge → 三路搜索 + 强制注入。counter 和 limits 必须出现——这是 challenge mode 的存在意义。
  • inspire → 三路搜索 + 降低阈值。主动发现排名低的隐藏关联——这是发现工具，不是搜索工具。
不是「既要又要」——是「不同 mode 要不同的东西」。（详见 architecture-decisions Q11）

Edges 存储格式：frontmatter 中 edges 以 JSON 字符串存储（含 confidence 字段）。Graph Walk 使用 avgConfidence × (1/pathLength)（对齐 agentmemory 评分公式）——不硬编码 per-type 分数。

Graph 无独立写盘文件——不需要。edges 的真相源在 wiki/*.md frontmatter 中——已经持久化了。buildGraph() 每次扫描 wiki 目录聚合 edges 到内存 GraphData。walkGraph() 在此内存结构上做图遍历。写一个 graph.json 副本反而引入 cache invalidation 问题（frontmatter 变了但 graph.json 没变 → 不一致）。agentmemory 同样不单独持久化 graph——它的 edges 存在 KV 数据库中，不是单独文件。产品定义 §08 原计划 graph.json 是 daemon 缓存——经过 v2.0 分析，决定不需要。

Graph 在两步中的角色不同（Q10）

边类型在三路搜索与 board 注入中扮演不同角色——这是 board 与三路「原理冲突、按 mode 分离」的具体落点：

三路搜索中的 GraphBoard 注入中的 Graph
作用提升排名（打分信号）保证存在（包含规则）
适用边类型derived_from / related / supportscontradicts / superseded_by
操作调整分数（被多 seed 引用 → 自然排前）调整集合（不管排名，必须出现）

代码实现：board.ts 的 forcedNodes 注入 switch 只处理 contradicts（→ counterNodes）和 superseded_by（→ relatedNodes）；derived_from / related / supports 不走注入，靠三路搜索的 walkGraph 自然提升排名。这避免了「假三路下靠注入兜底 related」掩盖结构信号的问题。

10 / OKF Integration

OKF 生态兼容（L5 Interface 层）

Open Knowledge Format (OKF) v0.1 是 Google Cloud 提出的厂商中立知识格式——用纯 Markdown + YAML frontmatter 表示知识。LiteWikiagent 的 wiki 节点格式天然匹配 OKF。

功能文件状态
导出okf/export.ts✅ 已实现——kind→type 映射 + createdAt→timestamp 映射 + body 渲染 + index.md/log.md 生成
导入okf/import.ts✅ 已实现——OKF bundle → wiki 节点 draft → 走 audit + gate 管线（不绕过质量闸门）
映射okf/mapping.ts✅ 已实现——9 种 WikiKind ↔ OKF type 双向映射
诚实评估：OKF 当前未解决任何实际问题。
OKF 导出做的事 = 字段改名（kind → type, createdAt → timestamp）。导入 = 逆向改名。
我们的 wiki 已经是纯 Markdown + YAML——git clone 就能分发，不需要 OKF 做中间格式。
OKF 生态尚未成形（v0.1 草案）——我们是第一个兼容的工具。
保留 okf/ 模块但不作为产品路线图优先级。已实现的三个文件（export/import/mapping）零维护成本——保留，等生态成熟后再激活。

在产品中的位置：OKF 属于 L5 Interface 层——不影响发动机设计。与搜索/图谱/审计的关系：导入节点走标准管线（被 qmd 索引、参与图谱、被审计）——OKF 不改变任何发动机逻辑。

11 / Counter Nodes

Counter 节点来源

当前 wiki/counters/ 为空。Counter 节点和 contradicts edge 是两个不同的东西——矛盾检测产生 edges，不产生 counter 节点。

来源机制状态
compile prompt在 compile prompt 中加一行指令：如果材料中有与主流矛盾或反例的命题，编译为 kind=counter🟢 最小改动，一行指令
contradiction.ts当同一主题出现 3+ 条 contradicts edges → daemon 调用 LLM 合成 counter 节点🟡 自进化未实现
inspire人类触发 inspire → 发现隐藏矛盾 → 手动标记为 counter✅ 人类可操作

优先方案：compile prompt。LLM 在读到材料时就能识别反方观点——比 daemon 事后从矛盾模式中合成更简单、更可靠。
