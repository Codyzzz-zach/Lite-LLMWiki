管线细化方案

基于 architecture-design v1.0 §04 和决策表 #6/#8 的精化。
目标：消灭 chunk 标记系统，统一为 prop-only 管线。

1 · 对齐后的共识

经过 Q1–Q4 四轮讨论，以下结论已确认：

#结论依据
A chunk 标记系统是设计违规残留。设计文档明确说「无 chunk 标记、无 fallback」，代码未完全执行。 architecture-design §04, 决策表 #6, #8
B 命题提取不影响语义理解。全文一次送 LLM，chunk 标记只是内联注释——不破坏语义流。 源码: proposition.ts → buildPropositionPrompt(chaseContent)
C DeepSeek 大上下文使 chunk 失去存在理由。chunk 存在是因为 v5 时代 context window 小——现在已经不是瓶颈。 DeepSeek V3: 128K input context
D 审计上下文是 chunk 唯一提供的合法功能——但可用 prop 窗口替代。±3 个邻近 prop 的原文 = 同等效果的上下文，不依赖 chunk 标记。 源码分析: selectChaseChunks() → selectPropsContext()
E compile 12000 字符截断是无设计依据的拍脑袋实现。正确做法是基于 maxTokens 反推安全输入上限。 源码: pipeline.ts → propTexts.slice(0, 12000)

2 · 发现的问题

P1 · selectChaseChunks 类型混淆 bug

semantic-audit.ts:166 — selectChaseChunks(config, sourceChase, propRefs)。函数签名期望 chunkRefs: number[]，但传入的是 prop 索引。e 基本画像有 8 个 chunk、104 个 prop——prop 50 → 查找 chunk 50（不存在）→ excerpts 空 → judge 无上下文。

P2 · collectPropIndices chunk fallback 设计违规

chase.ts:collectPropIndices() — 无 prop 时回退到 collectChunkIndices()。设计明确说「无 fallback」。

P3 · wiki-parser 解析 chunkRefs 设计违规

wiki-parser.ts:206-208 — 仍在解析 raw["chunkRefs"] 并转换为 propRefs。设计说「frontmatter 只用 propRefs」。

P4 · compile 硬截断 实现缺陷

pipeline.ts:propTexts.slice(0, 12000) — 12000 是拍脑袋的。实际安全上限应基于 maxTokens×3.5 计算（对 8192 输出 ≈ 28672 字符）。30 页论文的核心瓶颈在这里。

P5 · extract-props CLI 配置不一致 实现缺陷

extract-props.ts — model: "deepseek-chat" 硬编码，maxTokens: 16000，无 thinkingDisabled。pipeline.ts 使用 config.model + maxTokens: 32768 + thinkingDisabled: true。

3 · 设计决策

D1 · 消灭 chunk 标记 — chase 文件只用 prop 标记

Loader (store.ts) 写 chase 时不再生成 <!-- chunk:N -->。chase 文件 = frontmatter YAML + 正文 + <!-- prop N -->…<!-- /prop N -->。只有命题提取执行后才有 prop 标记——在此之前 chase 是纯文本。

D2 · 审计上下文改用 prop 邻近窗口

新增 selectPropsContext(content, propRefs, window=±3) 替代 selectChaseChunks()。功能：取 prop N 的原文，加上前后各 3 个 prop 的原文，拼接后作为 LLM judge 上下文。±3 窗口 ≈ 280 字——足够 judge 判断「claim 是否被原文支持」。

D3 · compile 截断改为自适应

propTexts.slice(0, 12000) → propTexts.slice(0, computeSafeInputLimit(maxTokens))。
computeSafeInputLimit(tokens) = tokens × 3.5 × (avgNodeChars / (avgPropsPerNode × avgPropChars))
对 8192 输出 tokens: 安全输入 ≈ 28672 字符。超出则分批 compile（按 prop 区间分批，跨批用 edges 连接）。

D4 · CLI 与 pipeline 统一 LLM 配置

所有 LLM 调用使用同一个 config.model（来自 .env）+ thinkingDisabled: true。删除硬编码 model 字符串。

D5 · 删除所有 chunk fallback 逻辑

删除：collectPropIndices 的 chunk fallback、wiki-parser.ts 的 parseChunkRefs、selectChaseChunks、readChaseChunks。
无 prop → 报错——不做 chunk 凑合。

4 · 目标数据流

PDF/TeX/MD → Loader: 纯文本正文 → chase: 正文 (无 chunk 标记) → extractPropositions (LLM 全文) → chase: 正文 + prop 标记 → compile (prop → wiki nodes + edges) → audit (prop 窗口上下文)

目标 chase 格式

---
sourceId: raw_pdf_e_的基本画像_3ef24c47e940cf85
sourceRoot: raw/original/pdf/e 的基本画像.pdf
---

e（欧拉数）约等于 2.718281828459045，是自然对数的底数。

<!-- prop 1 -->
1/e约等于0.3678794411，约等于36.8%。
<!-- /prop 1 -->

极限lim_{n→∞}(1-1/n)^n = 1/e。当n趋于无穷大时…

<!-- prop 2 -->
极限lim_{n→∞}(1-1/n)^n = 1/e。
<!-- /prop 2 -->

与当前格式的区别：

  • ❌ 无 <!-- chunk:1 --> / <!-- /chunk:1 -->
  • ✅ prop 用开闭标记环绕正文
  • ✅ 无 prop 标记的 chase 文件 = 纯文本（命题提取前的中间状态）
  • ✅ readChaseProps() 无需改：正则 /<!--\s*prop\s+(\d+)\s*-->/gi 仍然匹配
