01 / What We're Building

产品是什么

LiteWikiagent 是一个 agent 无关的可演化知识层。不是 RAG，不是笔记工具，不是搜索引擎。

产品定义

LiteWikiagent 是一个 agent 无关的可演化知识层： 它把原始材料编译成带审计信号的结构化 wiki， 给任何 agent 提供统一的知识操作基础设施——编译、审计、局面、维护。 agent 可以换，知识层的状态继承下来，wiki 随使用越来越准。

关键术语的精确含义：

agent 无关任何 agent 都可以接入——从 Claude Code 换到 Codex，知识层的审计信号、待确认队列、图谱状态全部继承。engine 不是特定 agent 的工具，是 agent 的外挂知识基础设施。
可演化知识层不只是一个静态 wiki 文件目录。是一个带有审计信号、board 装配策略、待确认队列、图谱关系的有状态系统。随使用变更准——旧错误被标记、新连接被标注、缺口被识别。
编译raw → chase 命题层 → wiki 节点的忠实转换。不是自由生成——每个 claim 锚定在命题级证据上。
审计LLM judge 对比 claim 与 chase 命题原文，给出 5 维度 verdict + 可信度评分。不是完美度量——是诚实声明：信号写入 frontmatter，告诉 agent 这条 claim 的可信度。配有三层防御防止审计错误。
局面wiki + 图谱 = LLM 的棋盘。board 装配 = 搜索（找相关节点）+ 强制注入（靠 kind、auditStatus、图谱 edges 确保关键棋子不被搜索遗漏）= 残局。6 种 mode 不是 prompt engineering——是 6 种不同的残局摆法。
越来越准旧错误被标记矛盾、人类确认后取代。新连接被 LLM 发现并标注候选、人类确认后写入 edges。知识缺口被主动标记。不是只增不减的堆叠——是信息卫生约束下的定向进化。
02 / The Engine

发动机：三个不可缺一的组件

LiteWikiagent 的发动机由三个组件构成：编译、审计、局面。缺一个——发动机就不成立。

用 AlphaGo 做类比：wiki + 图谱 = 棋盘，6 种 board 模式 = 残局的摆法，LLM = 下棋高手。发动机的核心价值不是"把文档变成文本"（谁都能做），而是"给 LLM 摆出一个经过审计的残局"（所有竞品都没做）。

审计 LLM judge 判断 claim 是否忠实于 chase 命题原文 wiki 节点 + chase 命题 auditStatus + auditScore 写入 frontmatter 这是产品第一壁垒。竞品没有语义审计——它们最多做断链检查和人在回路审批。engine 的审计是体系化的：命题级证据锚点 + 5 维度 verdict + 三层防御（evidence-anchored 输出、矛盾交叉验证、跨模型二次审计）。agent 自己做不了这种体系化审计。
局面 按 6 种 mode 装配上下文，交给 LLM 推理 wiki 节点 + 图谱 + audit 信号 结构化的 board → LLM answer 这是产品第二壁垒。竞品的查询是"关键词搜索 → 相关节点 → 扔给 LLM"。engine 的 board 是搜索 + 强制注入：搜索负责相关性，强制注入负责完整性——靠 kind、auditStatus、图谱 edges 确保 challenge 模式不遗漏 counter、trace 模式不遗漏 derived_from 链。这不是搜索优化的升级——是不同维度的信息补齐。
编译 + 审计 + 局面是一条链——去掉任何一个，另外两个的信息学职能就废了。 去掉审计 → chase 层是多余的（竞品不需要 chase 是因为不做审计）。去掉 chase → 审计没有证据锚点。 去掉 board → 审计信号无法被 query 有效利用。审计和局面共同构成产品的不可替代价值。
03 / System Boundary

系统边界：发动机 vs 外围

发动机从 chase MD 开始。之前的格式转换是可插拔的外围组件。

── 发动机边界 ── │ raw/ ──┬── PDF loader ──→ chase MD ──→ compile ──→ wiki ├── DOCX loader ─→ chase MD │ ├── EPUB loader ─→ chase MD ├── audit ──→ wiki + gate ├── MD loader ──→ chase MD │ └── TeX loader ──→ chase MD └── board ──→ answer │ │ 可插拔 loader 层 发动机本体
层什么调不调 LLM合同法
loader 层 格式转换——PDF/DOCX/TeX 转成纯文本 Markdown 不调 LLM，纯机械操作 接收 raw 文件 → 产出 chase MD（不带命题 marker 的纯文本）
daemon 命题提取 LLM 将 chase 纯文本分割为自包含的原子命题 调一次 LLM（每篇文档一次，结果永久缓存） 接收 chase 纯文本 → 产出带 prop marker 的 chase MD
发动机本体 编译、审计、board 装配、维护 compile 调 LLM · audit 调 LLM · query 调 LLM 接收带 prop marker 的 chase MD → 输出可审计 wiki + 残局
为什么边界画在这里：loader 层是可替换的——以后要加 EPUB loader、DOCX loader，加新的就行，发动机不用改。 竞品的 loader 和 engine 是耦合在一起的——这也是为什么它们能支持 30+ 格式而你不需要：你只要求 loader 遵守一个简单的合同（产出纯文本），剩下的命题提取由 daemon 统一处理。发动机的集中化设计让 loader 层可以无限扩展而不影响内核。
04 / Chase Layer

Chase 证据层：命题级分块

chase 不是纯格式转换——它是审计的证据基础设施。chunk 粒度为命题级，由 daemon 触发一次性 LLM 调用。

为什么是命题级 chunk

EMNLP 2024 (Chen et al., Dense X Retrieval) 的实验证据：命题级分块在检索 Recall@20 上比段落级提升 2-10%、比句子级提升约 6%。命题 = 自包含的最小事实单元——不能再拆成更小的独立命题，但也不需要额外上下文就能理解。

这对 engine 的含义：judge 对比 "claim X 是否被命题 7 支持" 时，输入是一个自包含的原子事实，没有无关上下文干扰。段落级 chunk 可能让 judge 在无关部分找到"看起来支持"的文字（false pass），残缺句子让 judge 读不懂上下文（false fail）。命题级在这两种误判上都是最优的。

命题提取由 daemon 触发

loader 产出 chase 纯文本（不带任何 marker）→ daemon 监听到新文件 → 调 LLM 做命题提取 → 在 chase 文件里插入 prop marker → chase 准备就绪。

这是一个一次性的、结果永久缓存的 LLM 调用。每篇文档调一次，compile 和 audit 都复用同一个命题编号。命题提取失败或质量差——daemon 写状态报告，agent 读到后决定下一步。

<!-- prop 1 --> Shor算法能在多项式时间内分解大整数。 <!-- prop 2 --> Shor算法对RSA加密构成威胁。 <!-- prop 3 --> 量子纠错需要大量物理量子比特。 <!-- prop 4 --> 当前量子计算机只有约1000个物理量子比特。

compile LLM 看到的是编号好的命题序列——它只需说 "claim X 来自命题 1 和命题 2"。audit judge 只需看编号对应的命题——两句话、一个事实、没有干扰。

chase 不是独有壁垒——"chase + 审计"组合拳才是。 竞品（如 nashsu）有两步 LLM 调用（分析 → 生成），但中间产物是 LLM 的语义理解（"关键实体有哪些"），不是持久化的证据锚点。 竞品不做审计，所以不需要 chase。engine 做审计，所以 chase 是前提条件。chase 的价值不是独立存在的——是和审计配套的系统属性。
05 / Compile

编译：Pro 两步提取 + Policy 确认 + 9 Kind Schema

编译是发动机的第一个组件——必要但非壁垒。核心设计是 Pro 深度思考 → Flash 结构化输出的分工。

Pro 两步提取

Step 1 (deepseek-v4-pro)：深度阅读 chase 命题序列，输出对材料主题、论点结构、关键关系的理解。Step 2 (deepseek-v4-flash)：基于 Step 1 的理解，将命题序列结构化提取为 wiki 节点——包括 frontmatter 字段、body sections、edges 候选。Pro 负责推理质量，Flash 负责 JSON 格式约束——同一次 ingest 里两种能力分工。

9 种 WikiKind · 固定 Schema

发动机 v1 使用固定的 9 种 kind 预设：concept（原子定义）、claim（断言+证据）、method（可复现过程）、case（具体实例）、equation（公式+解释）、question（开放问题）、insight（跨来源综合）、anchor（用户锚点直觉）、counter（反直觉视角）。kind 不是装饰性标签——它嵌在五个系统逻辑里（目录路由、编译 prompt、Policy 确认、Board 强制注入、图谱边类型）。换 kind 意味着五个地方都要改——v1 不做 Schema 定制。

远期方向：发动机代码不硬编码 kind 名称，从 Schema 配置文件读取——让不同领域的发动机实例可以有不同的 Schema 预设。

Policy 自动确认

编译产出的 proposition 需要确认才能进入 compile。三级策略决定自动确认的严格度：conservative（只确认 fact types + confidence≥0.7 + 有 evidence）、balanced（fact types 直接确认，soft types 需要 evidence）、expansive（几乎所有类型都确认，insight/counter/question 需要 limits）。不是入库价值判断——是编译质量闸门。

06 / Audit

审计：LLM Judge + 诚实声明 + 三层防御

审计是发动机的第一壁垒。engine 是所有竞品中唯一让 LLM 自动做语义审计的产品。

双重审查

结构审计（无 LLM）：检查 nodeId、kind、sourceChase、chunkRefs、Claim、Evidence——纯规则检查。结构不合格 → 阻止，exit 2。
语义审计（LLM judge）：5 维度判断——claim 是否被 chase 命题支持、是否引入原文没有的强判断、是否把解释写成事实、是否遗漏限制条件、evidence 是否真的支撑 claim。语义不合格 → 警告放行——不阻止，但必须告知。

三层防御

发动机的三层 LLM 调用（compiler 提取 → judge 审查 → query LLM 推理）存在系统风险：compiler 和 judge 可能共享认知偏差，对同一段原文犯同样的理解错误。这是 engine 独有的问题——因为竞品不做语义审计，不存在这个风险。用三层防御应对：

第一层 · 零额外成本

evidence-anchored 输出

judge 的 verdict 必须带原文锚点。不输出 "support=passed, score=0.85"，而是输出 "命题 3 第 2 句：'X 提升 Y 15%' → claim 忠实"。人类 review 成本从"重读整篇原文"降到"看一眼 judge 引用的那句话"。

第二层 · 图谱免费信号

矛盾检测交叉验证

节点 A 和 B 有 contradicts 边 → 至少一个的 audit 是错的。矛盾检测不告诉哪个错了，但它标注"这里有问题"——agent 或人类重点审查该区域。

第三层 · 按需付费

跨模型二次审计

auditScore > 0.9 且被 query/challenge 模式频繁引用的核心 claim——用不同模型（如 Claude）重新跑一次审计。两个不同模型犯同样错误的概率远低于同一个模型犯两次。不覆盖所有节点，只覆盖"高信任 + 高风险"节点。

诚实声明

审计结果写入 frontmatter：auditStatus + auditScore + judge 的 evidence anchor。query 的 board 中带节点的 audit 状态——LLM 能看到"这条 claim 的 semanticAudit 给了 0.6，judge 标注了 support 为 warning"。不假装 LLM judge 是客观度量——它是辅助信号，不是最终判决。系统的信任模型是"让错误可见"而不是"消灭错误"。

07 / Board

局面：搜索 + 强制注入 = 残局

Board 是发动机的第二壁垒。6 种 mode 不是 prompt engineering——是 6 种不同的残局摆法。

为什么要摆残局

竞品的查询模式是：混合搜索 → 找出最相关的节点 → 扔给 LLM。这个方案的问题是：搜索有盲区。搜索基于相似度——auditStatus=failed 的 tension 节点排名靠后、counter 节点和 claim 节点的相似度可能不高、derived_from 链上的节点可能没有共同关键词。给 LLM 一个空棋盘（只有相关性排名），LLM 可能漏掉关键棋子。

engine 的方案：搜索负责相关性（找最匹配的 seed nodes）+ 强制注入负责完整性（靠确定性字段确保关键棋子不遗漏）。两者组合 = 残局。

6 种 mode 的强制注入规则

mode搜索做什么强制注入什么注入的数据源（确定性）
ask混合搜索找 top seedslight——相关节点的 evidence + related + 少量 countergraph edges（related）
trace搜索 seeds全链追溯——derived_from 边上的所有祖先节点 + chase 命题原文graph edges（derived_from）+ chase prop marker
expand搜索 seedsmethods/cases/equations——不管搜索排名，强制带入所有相关 kind 的节点kind 字段
compare搜索 seeds按 sourceId 分组——不是搜出来的，是靠来源元数据强制分的sourceId frontmatter
challenge搜索 seedsALL counter + ALL limit——搜索可能全漏了，必须强制全部摆上graph edges（contradicts）+ kind（counter）
inspire搜索 seedstension nodes（auditStatus=failed）+ cross-kind relatedauditStatus + graph edges

强制注入依赖的数据源——kind、sourceId、auditStatus、图谱 edges——都是编译和审计阶段产生的确定性字段。强制注入的可靠性取决于这些字段的质量。图谱（P0 优先级最高的需求）是强制注入的升级——让 challenge 模式的 contradicts 边和 trace 模式的 derived_from 边不再依赖搜索，而是靠图谱的确定性关系。

为什么不是"简化成一个 mode + 让 LLM 自己判断"

如果只给 LLM 所有相关节点（方案 B），LLM 确实可能自己找出矛盾和追溯链。但搜索可能遗漏的关键节点——不在相关性排名前面的 counter、limit、tension——LLM 根本看不到。强制注入确保这些棋子出现在棋盘上。6 种 mode 不是限制 LLM 的推理——是补偿搜索的结构性盲区。

08 / Evolution

自进化：「越来越准」的五个方向

发动机不只是编译一次——它通过取代、自愈、连接、缺口让 wiki 随使用越来越准。

取代

新材料淘汰旧声明

新材料入库时与已有声明矛盾 → 系统标记 contradicts 边 → 旧声明标记为候选 superseded → 人类确认后取代生效。取代不是自动的——LLM judge 是主观的，自动取代有误杀风险。只标记不自动取代，裁决权在人类。

自愈

定期 lint + 分级修复

daemon 定期扫描：孤立节点、断裂引用、过时声明、缺失概念。机械问题自动修复（断链引用更新、图重建）。语义问题生成工单（内容改写、claim 更新）——等待人类确认而非自动修复。

连接

发现隐藏关联

daemon inspire 产出跨来源、跨 tag 的候选 edges。系统只负责发现和标注——写入 frontmatter edges 需要人类确认。LLM 发现的关联可能是"看起来有关但其实无关"的伪关联，人类必须做终判。

缺口

标记知识空白

系统知道 wiki 不覆盖什么——query 的 missingEvidence 标记、inspire 的 missingEvidence 类型、lint 的孤立节点检测。缺口被写入 progress.md，等人类决定是否填补。

强化

新材料支持已有声明时提升可信度

取代机制处理矛盾（contradicts），强化机制处理一致（reinforcement）。新编译节点与已有节点语义一致 → 标记 reinforcement 候选 → 人类确认 → auditScore 提升。不是遗忘曲线——是 evidence accumulation。知识不因不访问而贬值，但新证据让旧声明更可信。

回流：好答案写回 wiki

agent query 产生的答案如果 LLM judge 判断值得固化（有 fromWiki 引用、基于 board、有新 insight），daemon 标记为 reflow 候选。不自动写回——人类确认后才进入 wiki。回流节点标 reflowOrigin + derived_from 边指向推理来源节点。它们没有 chase 命题原文——audit 不能用于回流节点，信任层级天然低于编译节点。

强化：新材料支持已有声明时提升可信度

取代机制处理的是"新材料与旧声明矛盾"的场景（矛盾 → 标记 contradicts → 人类确认 → 取代）。但还有一个对称的场景：新材料支持了已有声明。当新编译的节点与已有 wiki 节点语义一致时（同样的 claim，来自不同来源），这说明已有节点的可信度应该提升——不是一个一次性评分，而是一个随 reinforcement 不断增强的信号。

关键区分：trivial reinforcement vs genuine reinforcement。如果新来源只是在复述同一个事实而没有提供独立的经验证据（如两本书都说"1/e ≈ 0.368"——这是一个数学常数，不存在"独立验证"），这种强化不应提升 auditScore——不是因为它错误，是因为它不增加可信度。只有当新来源提供了旧来源没有的独立验证（如"我们在实验中复现了这个结果"）时，才是 genuine reinforcement——应该确认并提升。daemon judge 在做一致性判断时必须输出两个维度：语义一致度（两个 claim 是否在说同一件事，0-1）和证据增量度（新来源是否提供了独立的经验证据，0-1）。只有当证据增量度超过阈值时，才标记为强化候选提交人类确认。

强化确认时人类看到的最小信息集：已有节点的 claim + 当前 auditScore + chase 命题锚点、新节点的 claim + source 信息 + chase 命题锚点、daemon judge 的一致性判断和增量判断、建议的新 auditScore。人类判断三个维度：语义一致性（真的在说同一件事吗？）、证据增量（新来源提供独立验证了吗？）、来源权威度（足够支撑提升吗？）。

这不是自动的——遵循信息卫生原则。daemon 只标记候选，人类确认后 auditScore 才更新。强化不修改节点内容，只更新 audit 元数据信号。这是"越来越准"的第五个方向：除了取代、自愈、连接、缺口——还有强化。

信息卫生原则

所有 LLM 二次生成的内容必须经过人类确认才能进入 wiki。这是防止 wiki 从"知识库"退化为"LLM 对话日志"的硬原则。
  • 可以自动做（机械操作）：格式转换、chunk 分割、结构审计、embedding 生成、增量检测、图重建、断链修复
  • 不能自动做（必须人类确认）：回流节点写入、LLM 发现的 edges 写入、语义 lint 修复、取代关系确认、任何 LLM 判断为"应该这样改"的内容变更
09 / Roles

三角色分工：daemon · agent · 人类

发动机的运行需要三个角色的明确分工。核心原则：daemon 是 CI 管道，agent 是决策者，人类是主编。

角色类比职责不可以做
daemon CI pipeline 监听文件自动命题提取、自动结构审计、定期跑 lint（只修复机械问题）、接收人类 inspire 命令（只产出候选）、回流候选标记。把结果写到状态文件（lint-report.json、progress.md）。 做语义判断、写回 wiki 内容、确认回流节点、确认取代关系、写入 edges——任何涉及"这个对不对"的决策
agent 开发者 + 运维 调用 CLI 做 compile、query、inspire。读取 daemon 状态文件，自己判断优先级——要不要处理这个矛盾、要不要告诉人类有回流候选。在 wiki 局面上做推理。每次会话开头读待确认清单，帮人类做预筛选。 替人类确认回流、替人类确认 edges——这些决策需要人的判断力
人类 主编 三件事：选材（放什么进 raw/）、终判（audit 结果是信号不是判决）、升级（daemon 和 agent 都处理不了的语义冲突）。定期批量处理待确认队列——不碎片化打扰。 日常操作——这些是 agent 和 daemon 的事

Daemon 七项职责

#职责触发daemon 决定什么留给 agent/人类决定什么
①命题提取新 chase 纯文本调 LLM 做命题提取命题提取质量是否需要人工复核
②文件监听自动导入chase 已有 prop marker触发 compile人类已经通过"放文件"做了选材决定。daemon 只执行
③wiki 变化自动审查wiki 文件变化触发 auditaudit 结果怎么用——agent 自己决定是否继续 query
④定期 lint + 分级修复定时器何时跑 lint、哪些机械问题可自动修复语义工单的优先级——agent 读 lint-report.json 后判断
⑤启发发现人类命令触发跑 inspire、产出候选（不写入 edges）哪个连接值得追查、是否写入 edges——人类触发命令 → daemon 执行 → 人类在结果中勾选确认
⑥回流候选标记定时器用 LLM judge 筛选候选是否写回——人类确认后才写入 wiki
⑦强化检测compile 后新节点语义一致时标记 reinforcement 候选是否提升已有节点 auditScore——人类确认后生效

人类确认的批量处理

人类确认不是通知驱动的碎片化操作——知识策展需要深度注意力。人类定期批量处理待确认清单（回流候选、edge 候选、语义 lint 工单、取代关系确认）。两个防护机制：

  • agent 预筛选：agent 在会话开头读待确认清单，用 LLM 按两个维度排序——风险（这条如果错了对 query 质量的影响）和置信度（daemon 判断的把握）。高风险 + 高置信度的排前面。低风险 + 低置信度的可以滞后。
  • 积压上限 + 自动降级：待确认数 < 20 → 正常积压。20-50 → agent 每次会话提醒人类。> 50 → 自动降级：低优先级项标记超时跳过，中项合并同类，只保留核心风险项。
10 / Competitive Moat

竞争壁垒：七个维度

在 6 个竞品中，LiteWikiagent 在以下 7 个维度上具有不可替代的差异。不是所有维度都是壁垒——编译不是，但审计和局面是。

🎯 Agent-agnostic 知识层

已构建

engine 是任何 agent 的外挂知识基础设施。agent 换了，审计信号、待确认队列、图谱状态全部继承。竞品的 agent 集成是单向工具调用——engine 是双向的（agent 调 engine 推理 + engine 回馈 agent 审计信号和待确认清单）。

⚖️ LLM Judge 语义审计

已构建

5 维度 verdict + 命题级证据锚点 + 三层防御。竞品没有语义审计——它们的"审查"是规则 lint 或人在回路审批。engine 是所有竞品中唯一让 LLM 自动做语义审计的产品。

♟️ Board 残局模型

已构建

6 种 mode = 搜索 + 强制注入。不是 prompt engineering——是补偿搜索结构性盲区的装配策略。所有竞品只有一种查询模式（搜索 + LLM）。

📋 Chase 证据粒度层

已构建

命题级 chunk 提供精确的证据锚点。竞品有"分析→生成"两步但中间产物是语义分析（不持久化、不做 chunk 级追溯）。chase 是审计的前提——没有 chase 就没有命题级证据对比。

🔧 Pro 两步 Ingest

已构建

deepseek-v4-pro 深度思考 → deepseek-v4-flash 结构化输出。分离推理质量和格式约束。nashsu 有类似两步但 Pro/Flash 的分工更精确。

🎛️ Policy 确认策略

已构建

conservative/balanced/expansive 三级自动确认。竞品要么全自动要么全手动——没有分级策略。基于 kind + confidence + evidence + limits 的多维过滤。

🤖 Daemon CI 管道

设计完成

daemon 做 CI 驱动的自动命题提取、lint、inspire、回标——不是对话驱动的。竞品依赖人在对话中触发维护操作。daemon 让自动维护成为系统属性。

7 个壁垒中 6 个已构建、1 个设计完成。这些能力是竞品很难复制的——不是技术难度问题，而是架构哲学差异。竞品要从"人类对话驱动"改为"agent-agnostic 知识层 + LLM judge 审计 + board 残局"需要重写整个产品模型。守在这 7 个壁垒上——这是差异化竞争的唯一正确策略。

12 / OKF Compatibility

OKF 生态兼容：零改造成本的格式标准化

Google Cloud 2026-06-12 发布的 Open Knowledge Format v0.1 与 LiteWikiagent 的 wiki 节点格式天然对齐。兼容 OKF 是零改造成本的生态占位。

OKF 是什么

OKF (Open Knowledge Format) 是 Google Cloud 提出的厂商中立知识格式规范——用纯 Markdown + YAML frontmatter 表示组织知识。核心理念：知识应该可以像数据一样被打包、被传递、被版本化管理。任何一个 OKF bundle 就是一个 Markdown 文件目录——不需要任何工具就能读，clone 一个 git repo 就能分发。

LiteWikiagent 天然兼容

OKF 要求LiteWikiagent 做法差距
UTF-8 Markdown + YAML frontmatter✅ 全量 wiki 节点就是这个格式零差距
type 字段（必填）kind 字段——语义等价映射关系：9 kind → OKF type
title + description已有零差距
tags已有零差距
timestampcreatedAt / updatedAt小映射
index.md + log.md已有零差距
Citations sectionchase 命题引用——细粒度远超 OKF向下兼容
Bundle 概念wiki/ 目录 = 一个 bundle零差距
Cross-linking (绝对/相对路径)[[wikilink]] + graph edgeswikilink 可映射到 Markdown link

兼容策略

不需要修改任何现有字段。只需：

  1. 导出功能：新增 llmwiki export --okf 命令——将 wiki/ 目录转换为 OKF 兼容的 bundle（kind → type 映射、createdAt → timestamp 映射、Chase 引用 → Citations section 映射）
  2. 导入功能：llmwiki import --okf <bundle>——读取 OKF bundle → 提取 concept 文档 → 进入编译管线（每个 concept 视为一个带完整 frontmatter 的节点 draft，走 audit + gate 流程）
  3. frontmatter 扩展：可选增加 okf_type 字段——不改变 kind 的语义，仅作为导出映射的显式声明
OKF 对 LiteWikiagent 的战略价值：不是"改用 OKF 格式"——是"可以声明 OKF 兼容"。这让你在开源生态中占据一个独特位置：唯一一个带语义审计的 OKF 兼容编译器。其他 OKF 工具只是格式转换器——你的发动机是"格式兼容 + 质量验证"。
13 / Search Infrastructure

搜索基础设施：qmd 集成

混合搜索是发动机 board 装配的必要能力。Karpathy V1 Gist 推荐的 qmd 提供了 BM25 + Vector + RRF + LLM 重排序的完整方案——比自己从零造混合搜索更轻量、更成熟。

qmd 是什么

qmd (Query Markup Documents) 是一个完全本地运行的文档搜索引擎。搜索管道：用户查询 → LLM 查询扩展 → 每个扩展查询并行执行 BM25 (SQLite FTS5) + Vector (本地 GGUF 模型) → RRF 融合 (K=60) → Top 30 候选 → LLM 重排序 (Qwen3-Reranker) → 位置感知混合评分 → 最终结果。总计约 2GB 本地模型，无需任何外部 API。

为什么用 qmd 而不是自建

维度自建混合搜索集成 qmd
BM25 后端需要集成 SQLite FTS5 或 Tantivy✅ 已内置，开箱即用
向量嵌入需要调 DeepSeek API 或本地模型✅ 本地 GGUF embedding 模型，零 API 成本
RRF 融合需要自己实现融合公式✅ RRF K=60 + Top-rank bonus 已实现
LLM 重排序需要额外 LLM 调用或自建✅ Qwen3-Reranker 本地重排序，yes/no + logprobs
CJK 中文支持需要做分词优化⚠️ 默认 embedding 为英文优化，需切换 Qwen3-Embedding
MCP 服务器需要自己搭建✅ 完整 MCP 服务器 (stdio + HTTP)，query/get/multi_get 工具
Agent 集成只支持 CLI✅ CLI --json 输出 + MCP + TypeScript SDK

集成架构

LiteWikiagent Board 装配 │ ├── seed nodes ← qmd search (BM25 + Vector + RRF + Rerank) │ │ │ └── 索引对象：wiki/*.md + chase/*.md │ ├── forced injection ← graph.json edges + kind/auditStatus │ └── final board → LLM query

引擎不绑定 qmd——qmd 是可替换的搜索后端。如果后续出现更好的方案或需要切换，engine 的 board 装配逻辑不变（只依赖搜索结果和强制注入的数据源）。qmd 的 CLI JSON 输出格式与 board 装配的输入接口对齐。

与自建的关系：qmd 处理搜索端（混合搜索 + 重排序），engine 处理 board 端（强制注入 + 局面装配）。两部分通过搜索结果（nodeId 列表 + relevance 分数）衔接。不是"用 qmd 替代自建"——是"用 qmd 处理搜索、engine 处理装配"，两者分工明确。
14 / Principles

不可动摇的设计原则

以下原则是产品定义的硬约束。违反任何一条 = 发动机设计不一致。

#原则含义
1 信息卫生 所有 LLM 二次生成内容必须经过人类确认才能进入 wiki。机械操作可以自动——语义操作不能。（见 §8 §9）
2 编译·审计·局面三者一体 发动机的三个组件不可分割。去掉任何一个，另外两个的信息学职能失效。编译是前提、审计是第一壁垒、局面是第二壁垒。（见 §2）
3 诚实声明 审计信号不是完美度量——是可信度辅助信号。LLM judge 是主观的，系统不假装客观。语义审计失败不阻止查询，但必须告知。错误要可见，不能隐形。（见 §6）
4 daemon 是 CI——不替 agent 做决策 daemon 负责执行和报告（发现 → 写状态文件），agent 负责判断和决策（读状态文件 → 定优先级）。daemon 的自动操作仅限于机械操作——不涉语义判断。（见 §9）
5 人类定期批量确认 人类确认不是碎片化通知——知识策展需要深度注意力。待确认清单积压到一定量后批量处理。agent 做预筛选排序。积压超过上限自动降级。（见 §9）
6 发动机与 loader 解耦 发动机本体不依赖任何特定 loader。loader 层是可插拔的——遵守一个合同（产 pure text chase MD）即可。命题提取由 daemon 统一处理——不在 loader 层调 LLM。（见 §3）
7 Board = 搜索 + 强制注入 搜索找最相关的节点，强制注入确保完整性（不依赖搜索排名的节点也上棋盘）。6 种 mode 是强制注入的不同配置。图谱是强制注入的升级——让 contradicts/derived_from 边不再依赖搜索。（见 §7）
8 命题级 chunk 是审计的前提 chase 层的 chunk 粒度为命题级（自包含的原子事实单元）。不是段落级、不是句子级——有 EMNLP 2024 论文的实验证据支撑。命题提取由 daemon 触发，一次性 LLM 调用，结果持久化。（见 §4）
9 回流节点信任层级低于编译节点 回流节点没有 chase 命题原文——audit 不能用于回流节点。它们的可信度天然低于编译节点。标 reflowOrigin + derived_from 边指向来源。Query 时 LLM 能区分"这是原文说的"和"这是 LLM 推的"。（见 §8）

本文档中所有关键设计决断的索引，每个决策标注了为什么选这个而不是另一个。

#决断为什么是这个对应章节
1发动机 = 编译 + 审计 + 局面（三者不可分割）编译是前提（非壁垒）、审计是第一壁垒、局面是第二壁垒。去掉任何一个，另外两个的信息学职能失效§2
2发动机边界在 chase 入口loader 层是纯机械的格式转换——可插拔、可扩展。发动机从 chase MD 开始，loader 和 engine 解耦§3
3chase chunk 为命题级EMNLP 2024 实验证据支持命题级 > 段落/句子级。命题级减少 judge 误判。一次 LLM 调用，结果持久化，compile 和 audit 复用§4
4命题提取由 daemon 触发（非 loader 层、非 compile 前）loader 层保持纯机械。daemon 集中处理命题提取逻辑。不对 compile 延迟造成影响§4 §9
59 种 WikiKind 固定 Schema（v1）kind 嵌在五个系统逻辑里——v1 不做 Schema 定制。v1 预设 9 kind，远期从配置文件读取§5
6语义审计 = 软关卡（警告放行）+ 三层防御LLM judge 是主观的——不应用主观判断做硬关卡。三层防御（evidence-anchored + 矛盾验证 + 跨模型二次审计）降低 judge 误判风险§6
76 种 Board mode = 搜索 + 强制注入只靠搜索 → 关键节点可能遗漏（counter、limit、tension 排名靠后）。强制注入靠确定性的 kind/auditStatus/edges 补齐搜索盲区§7
8所有 LLM 二次生成内容必须人类确认防止 wiki 从"知识库"退化为"LLM 对话日志"。机械操作可自动，语义操作不能自动§8 §12
9取代 = 只标记不自动取代LLM judge 是主观的——自动取代有误杀风险。裁决权在人类§8
10daemon 是 CI 管道（不替 agent 做决策）daemon 负责执行和报告，agent 负责判断和决策。类比 CI pipeline：自动跑测试，开发者决定修不修 bug§9
11人类确认 = 定期批量 + agent 预筛选 + 积压自动降级知识策展需要深度注意力——不适合碎片化通知。积压 > 50 自动降级——低优先项超时跳过，高优先项合并§9
12回流节点信任层级 < 编译节点回流节点没有 chase 命题原文——audit 不可用。可信度天然低于编译节点。标 reflowOrigin 确保 LLM 能区分来源§8
13不要遗忘曲线知识不因"很久没访问"变错。过时问题由矛盾检测处理。遗忘曲线增加复杂度不增价值——6/6 竞品都没有§8
14竞品借鉴重点：图谱 + 混合搜索 + 多 Provider + MCP这些都是可借的基础设施——竞品已有成熟方案。不是发动机的核心竞争力，但需要补齐以支持 board 强制注入和 agent 集成§11
15OKF 生态兼容（导出/导入）wiki 节点格式天然匹配 OKF——零改造成本。增加 llmwiki export --okf 和 import --okf 命令。战略价值：唯一带语义审计的 OKF 兼容编译器§12
16搜索基础设施用 qmd 而非自建qmd (Karpathy 推荐) 提供 BM25+Vector+RRF+LLM Rerank 的完整本地搜索方案。引擎通过 qmd CLI --json / MCP 接入。比自建混合搜索轻量 10x§13
17强化提升 auditScore新材料支持已有声明时（非矛盾、非取代）→ 标记 reinforcement 候选 → 人类确认后提升已有节点 auditScore。不是遗忘曲线（不要），是 evidence accumulation（要）§8
