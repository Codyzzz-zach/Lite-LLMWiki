AI agent systems tutorial

Loop Engineering：从“提示 AI”到“设计会自我推进的工作循环”

一份面向人类阅读的中文教程：用直白语言解释 loop engineering 是什么、它和 prompt/context/harness engineering 的关系、一个 loop 怎样运行、如何验证、在哪里容易翻车，以及如何从第一个小循环开始。

一句话定义

Loop engineering 是把“人不断提示 agent”的工作，改造成“系统按目标、状态和验证规则自动提示 agent、检查结果并决定下一步”的工程方法。

核心变化

你不再是循环里的手动时钟；你变成设计时钟、刹车、仪表盘和验收规则的人。

prompt 仍然重要，但它只是循环中的一个零件。真正的杠杆变成了：触发机制、上下文、工具权限、状态记忆、验证器、停止条件和人工升级路径。

01 / What it is

Loop engineering 到底是什么？

最容易理解的入口是一个熟悉场景：你让 coding agent 修测试。它改一点代码，你跑测试，失败了；你把错误贴回去，它再改；你再跑测试。看似 agent 在工作，实际上你在做循环控制：决定什么时候开始、把什么信息喂回去、什么时候算成功、什么时候放弃。

Loop engineering 做的事，就是把这套重复控制结构显式设计出来。系统会自己发现待办事项，生成下一条提示，调用 agent 或子 agent 执行，读取真实环境反馈，把状态写到外部记忆里，再根据验证结果决定继续、重试、回滚、提交给人类，或停止。

因此，loop engineering 不是“写一个更厉害的 prompt”。它是把 prompt 放进一个可重复、可观察、可终止的控制系统里。prompt 回答“这一步怎么说”，loop 回答“什么时候说、对谁说、带什么上下文、拿什么证据验收、失败后怎么处理、成本上限是多少”。

一个有用的判断标准：

如果每一步都需要你读结果、复制错误、输入下一句，那你就是 loop。如果系统能自己读反馈、决定下一步，并在达到可验证条件后停下，那你设计了一个 loop。

  Human["人类逐轮提示"] --> Agent["Agent 执行"]

02 / The stack

它不是替代 prompt engineering，而是站在更高一层

Prompt、context、harness、loop 不是互相淘汰的流行词，而是从小到大的工程边界。越往上，系统越自主，错误也越容易累积，所以验证和约束也要越硬。

层级 关心什么 核心问题 典型失败
Prompt engineering 单次输入怎么写 我要对模型说什么？ 指令含糊、例子不足、语气或角色不对
Context engineering 当前窗口里应该放什么 模型此刻需要看见哪些文件、历史、规则和摘要？ 上下文噪声、过期信息、关键信息缺失、窗口膨胀
Harness engineering 单次 agent run 的环境 给它哪些工具、权限、恢复策略和完成定义？ 工具太危险、权限过大、失败恢复混乱、done 定义不清
Loop engineering 多次 run 组成的自推进循环 怎样让它自己启动、分派、验证、记忆、停止或升级？ 无限重试、验证债、理解债、成本失控、错误自我放大

03 / One turn

一个 loop 的五个动作

把一次循环拆开，其实就是五件事：发现值得做的事、交给执行者、验证结果、保存状态、安排下一次。缺少任何一个，loop 要么转不起来，要么只是在原地消耗 token。

Discovery：发现工作

从 CI 失败、issue、日志、邮件、报表、监控告警或 backlog 中找出值得处理的事项。关键不是“读数据”，而是判断哪些事现在值得做。

Handoff：隔离交接

把任务交给 agent 或子 agent，最好放进独立 worktree、分支或沙箱，避免多个 agent 同时改同一片工作区。

Verification：验证结果

运行测试、lint、类型检查、构建、审查器或业务规则。最可靠的是命令式证据；无法机械验证时，再用模型评审补足。

Persistence：保存状态

把发现、尝试、结论、失败原因和下一步写入 markdown、数据库、工单或看板。模型会忘，外部状态不会忘。

Scheduling：决定下一步

根据验证结果选择继续、重试、回滚、开 PR、归档、等待下次触发，或升级给人类。

04 / Components

六个组成件：heartbeat、body、memory 和 checker

不同工具的叫法会变，但可靠 loop 的结构很稳定：有心跳，有隔离，有可复用知识，有外部连接，有 maker/checker 分工，还有一条贯穿始终的状态脊柱。

组成件 作用 简单例子 设计要点
Automation / heartbeat 让 loop 自动启动 每天 9 点检查 CI；PR 打开时自动审查 选择合适频率，设置成本和时间上限
Worktree / isolation 让并行 agent 不互相覆盖 每个修复任务一个 git worktree 隔离分支、明确合并策略、避免共享脏状态
Skills / reusable knowledge 把项目规则固化下来 测试命令、代码规范、发布流程、禁改区域 短、准、可维护；用名称触发，不粘贴长指令
Connectors / plugins 连接真实工具 GitHub、Linear、Slack、Gmail、数据库、监控 最小权限、审计日志、危险动作需人工确认
Sub-agents 拆分执行、审查、总结等角色 一个 agent 写代码，另一个 agent 做审查 maker 和 checker 分离，避免自己给自己判满分
External memory 跨运行保存状态 TODO.md、progress.md、工单状态、PR 备注 记录尝试过什么、为什么失败、下一次从哪里继续
不要把 memory 误解成聊天记录。

长对话会积累旧推理、错误假设和过期文件片段。更稳的方式是把状态写到外部：任务清单、日志、测试结果、分支、PR、工单。每次新运行只读必要状态，保持上下文干净。

05 / Patterns

常见 loop 模式：从最简单到可并行

Loop engineering 不要求一开始就搭大型多 agent 系统。真正成熟的做法通常是从小而可验证的 loop 开始，再逐步增加自动化、隔离和并行能力。

模式 适合场景 优点 风险
ReAct loop 需要边行动边观察反馈的任务 简单、通用，是多数 agent loop 的基础 没有硬停止条件时容易绕圈
Ralph-style fresh context 按任务清单逐项完成、每轮可独立验证 每轮重置上下文，减少长会话腐化 依赖外部任务文件质量；粒度太大时不稳
Plan-and-execute 多阶段迁移、复杂重构、批处理 规划和执行分离，减少中途漂移 计划过早僵化，可能无法适应新反馈
Evaluator-optimizer 有明确验收标准的生成与改进任务 maker/checker 循环明确，适合迭代优化 评审器标准弱时，系统会学会讨好评审器
Orchestrator-workers 可拆分、可并行的大型任务 多个 worker 用独立上下文并行推进 协调、合并、审查成本上升 
# 一个极简 Ralph-style loop 的思想，而不是生产建议
while not done:
    start_fresh_agent()
    agent.read("PLAN.md", "STATUS.md")
    agent.pick_one_small_task()
    agent.implement()
    run_tests()
    agent.update("STATUS.md")
    stop_this_agent()

# 1. 清晰的小任务
# 2. 可验证的成功条件
# 3. 外部状态文件
# 4. 每轮干净上下文
06 / Verification

真正困难的不是“让它做”，而是“让它知道何时该停”

一个 loop 的质量上限，往往由 checker 决定。执行者可以很强，但如果验收条件含糊，loop 会把模糊目标解释成“差不多完成”，或者在无法达成的目标上无限重试。

Rule 1

把目标写成可证明的结束状态。不要写“优化登录模块”，要写“登录相关测试全绿，lint 通过，公共 API 不变”。

Rule 2

优先使用确定性证据。测试、编译器、类型检查、lint、覆盖率、截图 diff、数据库约束，比模型自评更可靠。

Rule 3

永远设置天花板。最大轮数、最大耗时、最大花费、连续无进展次数，至少要有一种硬停止。

字段 弱写法 强写法
结束状态 把 checkout flow 做好一点 所有 checkout E2E 测试通过，并新增一个覆盖优惠码失败路径的测试
证据 看起来没问题 npm test -- checkout 和 npm run lint 均退出 0
约束 尽量别影响别的地方 不得修改 public API，不得删除现有测试，不得触碰支付密钥配置
预算 一直做直到完成 最多 12 轮；同一错误连续出现 3 次就升级给人类

  M --> E["环境证据

  V -->|通过| S["停止：提交结果"]

07 / Risks

Loop engineering 不会消除工程责任，只会放大设计质量

一个好 loop 会放大你的判断；一个坏 loop 会放大你的含糊、懒惰和权限设计错误。越是能无人值守，越需要更强的边界、审计和人工回路。

验证债 没有可靠 checker 的 loop 会持续制造“看似完成”的结果。解决方法是把验收条件工程化，而不是依赖 agent 自信。
理解债 loop 交付越快，你越可能不理解仓库里发生了什么。需要定期读 diff、看日志、复盘失败，而不是只看“通过”。
上下文腐化 长循环会把错误路线和旧输出塞满窗口。用外部状态、摘要、子任务隔离和 fresh context 来减压。
成本失控 定时任务、子 agent、评审器会迅速叠加 token。先低频、只读、可观测，再逐步开放写权限。
权限过大 能连接 GitHub、Slack、数据库或生产系统的 loop 必须有最小权限、审计日志和危险动作确认。
认知外包 最舒服的失败是停止思考。loop 应该替你做重复劳动，不应该替你拥有判断力。
08 / Start small

如何设计你的第一个 loop

第一条 loop 不要自动改生产代码。选一个重复、低风险、结果可检查的任务，让它先只读、只总结、只写状态文件。等它稳定产生你愿意采纳的结果，再让它开分支、写草稿、跑验证。

选一个重复任务

例如每天整理 CI 失败、每周汇总依赖更新、每月生成报告草稿、每天筛选重要邮件。

写清触发条件

是每天 9 点、PR 打开、CI 失败、收到特定邮件，还是手动启动？触发越明确，loop 越可控。

写成可验证的 done

用“没有未分类文件”“所有测试通过”“每个 PDF 都有摘要”这样的状态，而不是“整理好”“改进一下”。

限制工具和权限

第一版尽量只读。需要写入时，先只写 markdown 或开草稿 PR，不自动合并、不自动删除、不碰生产。

加入 checker 和天花板

指定测试命令、计数规则、审查 agent 或人工确认，并设置最大轮数、最大耗时或最大花费。

记录并复盘

让 loop 写下它看到什么、做了什么、为什么停下。最初几天你要读日志，修正触发、上下文、验收和权限。

成熟度 Loop 做什么 人类还在哪里
Level 0：手动 你逐轮提示 agent 每一步都在
Level 1：只读 triage 定时读取信息，写 findings 到文件或看板 你决定是否行动
Level 2：草稿分支 在隔离 worktree 里起草修复或报告 你审查每个 diff
Level 3：验证后 PR maker 写，checker 验，通过才开 PR 你批准合并
Level 4：低风险自动化 对依赖补丁、格式化、简单 flaky retry 等低风险任务自动合并 你审计日志和抽查结果
第一个练习模板：

“当 触发条件 发生时，读取 输入来源，完成 小而明确的任务，把状态写到 外部记忆。当 可验证条件 成立时停止；如果 失败/无进展/超预算，升级给我。”
