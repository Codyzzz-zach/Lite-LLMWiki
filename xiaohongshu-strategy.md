# LiteWikiagent 小红书运营策略完整方案

> 基于产品深度分析，为 LiteWikiagent 打造小红书生活方式品牌

## 一、产品定位分析

### 1.1 产品核心本质

LiteWikiagent 是一个 **agent-first 个人知识编译器**：

- **用户选材** → 系统忠实编译 → Wiki承载结构 → LLM在局面中推理 → Agent按协议执行
- 把 PDF/Markdown/TeX 原始知识文件，编译成可溯源的 Markdown Wiki
- 关键差异化：**语义忠实度审查（Semantic Audit）** + **局面式查询（Query Board）**

### 1.2 小红书定位策略：从 CLI 工具到生活方式品牌

在小红书上，不能直接卖 CLI 命令行工具。需要将技术产品翻译成**生活方式叙事**：

| 技术概念 | 小红书翻译 |
|----------|-----------|
| 知识编译器 | "给你的知识做一次编译——像代码一样运行你的阅读" |
| 第二大脑 | "我建了一个AI第二大脑，论文读一遍就够了" |
| Semantic Audit | "AI帮你审——每条知识都能追溯原文，不再被幻觉骗" |
| Query Board | "6种提问姿势，让你的第二大脑不止于搜索" |
| Chase层溯源 | "知识有出处，每句话都知道它从哪来的" |
| Agent Contract | "让AI替你读书，但它必须先过审查" |

**一句话小红书定位**：

> 🌿 LiteWikiagent — AI时代的知识编译器，把你的阅读编译成可溯源的第二大脑

### 1.3 品牌性格设定

基于产品已有的品牌视觉系统（Forest Green #0D5C41 + Compile Green #00D97E + Surface Mint #F0FDF4），品牌性格：

- **基调**：安静森林中的编译之光 — 不喧嚣，有深度
- **人设**：一个懂 AI 又懂人文的知识建筑师，不是技术推销员
- **语气**：轻学术、重体验，偶尔冷幽默，从不爹味说教
- **关键词**：可溯源 · 忠实编译 · 知识美学 · 效率哲学

### 1.4 目标用户画像（小红书版）

**核心四圈层**：

1. **研究者 / 研究生**（最核心）
   - 痛点：论文太多读不完，读完记不住，记住了找不到出处
   - 场景：每周精读2-3篇论文，需要构建可追溯的知识库
   - 小红书标签：#研究生日常 #论文精读 #科研效率

2. **AI 开发者 / Prompt 工程师**
   - 痛点：想让 Agent 真正基于自己的知识工作，但 LLM 总是跑偏
   - 场景：构建 Agent 使用的知识基底，语义审查保证忠实度
   - 小红书标签：#AI开发 #Agent开发 #DeepSeek #Prompt工程师

3. **知识管理爱好者**
   - 痛点：Obsidian/Notion 用久了还是松散笔记，缺结构、缺追溯
   - 场景：追求"第二大脑"的真正可信赖结构，不是随意联想
   - 小红书标签：#知识管理 #第二大脑 #Obsidian #Notion替代

4. **效率美学追求者**（泛流量入口）
   - 痛点：信息焦虑，想找到一种"读了就不忘"的方法
   - 场景：被效率工具/学习方法吸引，愿意尝试新东西
   - 小红书标签：#学习方法 #效率工具 #信息管理 #自我提升


### 2.3 小红书文案风格指南

**标题公式**（小红书算法友好）：

```
公式A: 数字 + 痛点 + 解法
  "读100篇论文还是忘？我给知识做了一次编译"

公式B: 情感钩子 + 差异化
  "我把论文编译成第二大脑，AI再也不胡说了"

公式C: 美学悬念 + 惊喜
  "一篇PDF编译成9个知识节点，看着结构长大的感觉"

公式D: 对比冲击
  "同一问题：普通ChatGPT vs 用第二大脑查询的回答"
```

**正文风格**：

- 开头3行：情感共鸣或痛点触发，不讲技术
- 中段：用"故事/体验"带出功能，不是功能说明书
- 结尾：一句轻学术的哲学收束 + CTA（"试试把你的下一篇论文编译进第二大脑 → GitHub链接"）
- 禁止：爹味说教、技术名词堆砌、没有场景的功能介绍

**首图风格**（与品牌视觉一致）：

- 主色调：Forest Green (#0D5C41) + Compile Green (#00D97E) + Surface Mint (#F0FDF4)
- 图片类型优先级：知识节点可视化图 > 终端界面美学截图 > 数学概念插画 > 手绘笔记风格
- 字体：Geist / SF Pro 系列等宽 + 无衬线组合
- 禁止：花哨渐变、3D特效、低信息量的纯色背景

### 4.3 社区互动策略

**互动优先级**：

1. **发布后2小时**：回复所有评论，用追问引导深度讨论
2. **每日30分钟**：在相关话题下评论5-10篇热门笔记（知识管理/AI/论文精读圈）
3. **每周1次**：发起互动型笔记（"你在读什么论文？"、"你的知识管理工具是什么？")
4. **实时关注**：被收藏/关注后私信感谢 + 引导体验

**评论回复模板**（不是机械模板，是思路参考）：

```
类型A: 痛点共鸣型
  用户: "论文读完就忘，太真实了"
  回复: "核心问题不是记忆，是编译——你读的时候缺一个'结构提取层'。试试把下一篇论文丢进编译器看看"

类型B: 工具对比型
  用户: "我用Obsidian也记笔记啊"
  回复: "Obsidian记的是松散笔记，编译器产出的是有Claim/Evidence/Limits的结构节点，而且每条都能追溯原文页码。两个层次的工具"

类型C: 好奇探索型
  用户: "这个怎么用？"
  回复: "最简单的方式：把一篇PDF丢进去，一条命令就编译好。GitHub开源的，可以试试"
```

## 六、转化路径设计

### 6.1 从小红书到 GitHub 的转化链路

```
小红书笔记 → 评论区引导 → GitHub README → 试用安装 → 深度用户 → 社区贡献者

每一步的引导:
  Step1: 笔记结尾 "开源在GitHub → 链接在评论区置顶"
  Step2: 评论区置顶评论包含完整GitHub链接 + 一句核心卖点
  Step3: GitHub README要有小红书入口（"来看看用户怎么用LiteWikiagent编译他们的论文"）
  Step4: 首次试用建议用conservative策略，降低门槛
  Step5: 试用后引导加入GitHub Discussions / 微信社群
```

### 6.2 转化 CTA 设计

**笔记内CTA**（自然嵌入，不强推）：

```
软CTA: "试试编译你的下一篇论文 → 开源免费"
中CTA: "GitHub搜LiteWikiagent，一条命令就能开始"
硬CTA（仅产品展示笔记）: "安装教程在评论区置顶，5分钟上手"
```

## 八、风险与应对

### 8.1 主要风险

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| CLI工具在小红书受众偏窄 | 流量天花板低 | 用"知识生活方式"叙事扩大受众，CLI只是实现手段 |
| 技术内容容易枯燥 | 互动率低 | 70%内容必须是故事/体验/美学，技术只做10% |
| AI幻觉话题热度衰退 | 热点内容失效 | 准备替代热点储备（知识管理趋势/新论文/效率方法论） |
| 竞品跟进类似概念 | 差异化模糊 | 强化"语义忠实度审查"和"局面式查询"的独特性 |
| 开源项目转化路径长 | 从看笔记到试用门槛高 | 降低首次体验门槛：准备一键Demo脚本 |

### 8.2 内容红线

- 不夸大产品能力（当前v5仍有已知限制）
- 不暗示"取代ChatGPT/Obsidian"（定位是互补，不是替代）
- 不在笔记中直接贴长段代码（小红书用户不读代码）
- 不做"读了这篇就不用读论文"的误导暗示
- 不用焦虑营销（"你再不建第二大脑就落后了"）
